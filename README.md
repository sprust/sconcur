# SConcur

Библиотека конкурентности для PHP поверх собственного расширения на Go. PHP-фаза
(Fiber) приостанавливается, пока Go-расширение выполняет задачу (операции MongoDB,
sleep) конкурентно в горутинах. Обмен между PHP и Go идёт через MessagePack.

## Идея

Суть SConcur — **вынести I/O-операции (MongoDB, sleep и т.п.) в Go-часть и выполнять
их параллельно**. PHP по своей природе синхронен: запросы к БД блокируют процесс по
очереди. Здесь же каждая I/O-операция уходит в Go и исполняется в отдельной горутине,
поэтому десятки запросов летят «веером» и общее время ограничено самой медленной
операцией, а не их суммой. PHP остаётся тонким слоем-оркестратором, вся конкурентность
живёт в Go.

### Почему Go

- **Очень удобная модель конкурентности.** Горутины + каналы дают дешёвый параллелизм
  «из коробки»: одна задача = одна горутина, результаты собираются через канал. Не нужны
  внешний event-loop, пул потоков или callback-ад.
- **Простота добавления новых фич.** Чтобы подключить новую I/O-операцию, достаточно
  написать обычный синхронный Go-обработчик — рантайм сам выполнит его конкурентно.
  Блокирующий драйвер (например, официальный MongoDB-драйвер) используется как есть,
  без переписывания под асинхронность.
- **Готовая экосистема.** Зрелые драйверы и библиотеки Go переиспользуются напрямую.

## Применение и ограничения

- **Только CLI.** Библиотека рассчитана на долгоживущие CLI-процессы (воркеры, демоны,
  скрипты, консольные команды). **С PHP-FPM использовать нельзя** — расширение держит
  Go-рантайм и горутины на уровне процесса, а модель FPM (короткий запрос-ответ, общие
  процессы пула) этому противоречит.
- **Нельзя `pcntl_fork` после загрузки расширения.** Go-рантайм и его горутины не
  переживают `fork`: дочерний процесс получит неработоспособный рантайм (зависания,
  падения). Если нужен пул воркеров — форкайтесь **до** первого обращения к расширению,
  либо запускайте отдельные процессы (`exec`), а расширение инициализируйте уже в
  потомке.
- **Только NTS (non-thread-safe).** ZTS-сборка PHP не поддерживается.
- **Не вызывайте `exit()`/`die()` при активных задачах.** Если оборвать процесс, пока
  в Go выполняются задачи (незавершённый `WaitGroup`, открытый курсор), поведение не
  определено: результаты и освобождение ресурсов не гарантируются. Сначала доведите
  задачи до конца или остановите их (`WaitGroup::stop()`), затем завершайте процесс.
- **Конкурентный режим необязателен.** Любую фичу, которая ходит в Go-часть
  (`Sleeper`, `Collection` и т.д.), можно вызывать и **вне** `WaitGroup` — как обычный
  синхронный вызов. Вне Fiber `FeatureExecutor` определяет неасинхронный контекст и
  просто дожидается результата (`Extension::wait`), возвращая его сразу. Это удобно,
  когда конкурентность не нужна, а единый API хочется сохранить.

```php
// синхронно, без WaitGroup — вернёт результат сразу
$collection->insertOne(['name' => 'example']);
```

## Принцип работы

`WaitGroup` — это драйвер-цикл поверх PHP Fibers. Каждый замыкание-таск оборачивается
в `Fiber`; когда внутри фазы вызывается асинхронная фича, фаза **приостанавливается**
(`Fiber::suspend()`), а соответствующая задача уходит в Go и выполняется в отдельной
горутине. При ожидании PHP через расширение (`Extension::wait` → cgo `wait` →
`Flow.Wait()`) слушает Go-канал `results`, в который горутины фич пушат свои
результаты. `WaitGroup` забирает их по мере готовности (в произвольном порядке) и
возобновляет ту фазу, чья задача завершилась.

### Схема: PHP Fiber ↔ Go goroutine

```
 PHP                                                 │ Go (расширение)
                                                     │
 WaitGroup::add(fnA) → Fiber → start()               │
   Sleeper::sleep() → FeatureExecutor::exec()        │
     Extension::push(flow, taskA) ──────────────────►│ Handler.Push → Flow.HandleMessage
   Fiber::suspend()  (управление возвращается)       │   go handler.Handle(taskA): sleep (выполняется)
                                                     │
 WaitGroup::add(fnB) → Fiber → start()               │
   Collection::insertOne() → exec()                  │
     Extension::push(flow, taskB) ──────────────────►│ Handler.Push → Flow.HandleMessage
   Fiber::suspend()                                  │   go handler.Handle(taskB): insert (выполняется)
                                                     │ (горутины A и B выполняются параллельно)
 WaitGroup::iterate()                                │
   FeatureExecutor::wait(flow) ─────────────────────►│ Flow.Wait() блокируется на канале results
   resultB ◄─────────────────────────────────────────│ insert завершился первым → result(taskB)
   $fiberB->resume(resultB) → yield keyB             │
   FeatureExecutor::wait(flow) ─────────────────────►│ Flow.Wait()
   resultA ◄─────────────────────────────────────────│ sleep завершился → result(taskA)
   $fiberA->resume(resultA) → yield keyA             │
                                                     │
   stop() → stopFlow(flow) ─────────────────────────►│ Flows.DeleteFlow → Flow.Cancel (ctx)
```

Результаты приходят в порядке завершения задач, а не в порядке `add()`.

### Слои и поток вызовов

```
┌───────────────────────── PHP  (src/) ──────────────────────────┐
│                                                                │
│ WaitGroup ──► State        реестр Fiber ↔ flow ↔ task          │
│     │                                                          │
│     │  тело замыкания-таска                                    │
│     ▼                                                          │
│ Features  (Sleeper, Mongodb\Collection, …)                     │
│     │                                                          │
│     ▼                                                          │
│ FeatureExecutor ──exec/next/wait──► Connection\Extension       │
└────────────────────────────────────────────────────────────────┘
                          │  ▲
         push/wait/next ──►│  │── результат (MessagePack)
          (cgo, msgpack)   ▼  │
┌────────────────────────── Go  (ext/) ──────────────────────────┐
│                                                                │
│ main.go (cgo exports) ──► Handler ──► Flows ──► Flow           │
│                                                 │              │
│                                                 ▼              │
│                          Task (горутина) ──► sleep / mongodb   │
│                                 │                              │
│                                 ▼                              │
│                          results channel ──► Flow.Wait()       │
└────────────────────────────────────────────────────────────────┘
```

Ключевые сущности:

- **`WaitGroup`** — публичный API: `add()`, `iterate()`, `waitAll()`, `waitResults()`.
  Каждый экземпляр владеет уникальным `flowKey`.
- **`State`** — статический реестр связей `Fiber ↔ flow ↔ task`.
- **`FeatureExecutor`** — точка входа для фич; определяет async-контекст через
  `Fiber::getCurrent()`, отправляет задачу в Go и приостанавливает фазу.
- **`Connection\Extension`** — синглтон-обёртка над экспортированными C-функциями
  Go-расширения (`push`, `wait`, `next`, `stopFlow`, `destroy`).
- **Go: `Handler → Flows → Flow → Task`** — на каждый `flowKey` создаётся один `Flow`
  с каналом результатов; каждая задача исполняется в своей горутине, результат
  отправляется в общий канал флоу.

### Жизненный цикл одной задачи

1. `WaitGroup::add($callback)` оборачивает замыкание в `Fiber`, регистрирует связь
   `fiber → flow` в `State` и вызывает `$fiber->start()`.
2. Фаза выполняется синхронно до первого асинхронного вызова. Внутри фичи
   (`Sleeper::sleep`, `Collection::insertOne`, …) вызывается
   `FeatureExecutor::exec($method, $payload)`:
   - `State::getCurrentFlow()` определяет, что мы внутри зарегистрированной фазы
     (`isAsync = true`);
   - `Extension::push()` формирует `taskKey = flowKey:counter` и через cgo `push`
     отправляет задачу в Go;
   - связь `task → fiber` сохраняется в `State`;
   - `Fiber::suspend()` возвращает управление вызывающему коду.
3. Если фаза завершилась **не приостановившись** (синхронный таск), её результат
   складывается в `syncResults`. Иначе живая фаза паркуется в `$fibers` по `fiberId`.
4. На стороне Go `push → Handler.Push → Flows.InitFlow → Flow.HandleMessage` создаёт
   `Task` и запускает **горутину** с обработчиком фичи. Результат уходит в канал
   `Flow.results`.
5. `WaitGroup::iterate()` (генератор-драйвер) сначала отдаёт `syncResults`, затем в
   цикле вызывает `FeatureExecutor::wait($flowKey)` → cgo `wait` → `Flow.Wait()`,
   который **блокируется до завершения любой** задачи флоу и возвращает её результат
   (MessagePack).
6. По `taskKey` находится `fiberId` (`State::pullFiberByTask`), и
   `$fiber->resume($taskResult)` возобновляет фазу: `Fiber::suspend()` внутри
   `FeatureExecutor` возвращает `TaskResultDto`, фаза продолжается.
7. Если фаза завершилась — `iterate()` отдаёт `callbackKey ⇒ <return value>`. Если фаза
   снова приостановилась (например, курсор запросил следующий батч через `next`), цикл
   продолжается. По завершении `finally → stop()` очищает `State` и Go-флоу.

`waitAll()` — это `iterator_count(iterate())`; `waitResults()` собирает результаты в
массив по `callbackKey`.

## Build
```shell
rm -f build/sconcur.so build/sconcur.h && \
  CGO_CFLAGS=$(php-config --includes) \
  go build -buildmode=c-shared -o build/sconcur.so .
```
## echo test
```shell
php -d extension=./build/sconcur.so -r "echo \SConcur\Extension\ping('hello') . PHP_EOL;"
```
## example
```php
$sleeper = new \SConcur\Features\Sleeper\Sleeper();

$collection = new \SConcur\Features\Mongodb\Connection\Client('mongodb://localhost:27017')
    ->selectDatabase('example')
    ->selectCollection('example');

$waitGroup = \SConcur\WaitGroup::create();

$waitGroup->add(
    function () use ($sleeper) {
        $sleeper->sleep(seconds: 1);

        return 1;
    }
);

$waitGroup->add(
    function () use ($sleeper) {
        $sleeper->usleep(milliseconds: 11);

        return 2;
    }
);

$waitGroup->add(
    function () use ($collection) {
        $collection->insertOne(['name' => 'example']);

        return 3;
    }
);

$waitGroup->add(
    function () use ($collection) {
        $iterator = $collection->aggregate([
            [
                '$match' => ['name' => 'example'],
            ],
        ]);

        foreach ($iterator as $item) {
            echo $item['name'] . PHP_EOL;
        }

        return 4;
    }
);

$iterator = $waitGroup->iterate();

foreach ($iterator as $key => $item) {
    echo "result: $item" . PHP_EOL;
}
```

## Планы

- **Упростить и описать добавление новых фич.** Сейчас новая I/O-операция требует
  согласованных изменений на обеих сторонах (PHP-обёртка + Go-обработчик + enum'ы методов и
  команд + сериализация payload'а). Нужно свести это к минимальному, документированному
  контракту: понятная точка расширения, шаблон «как добавить фичу» и пример, чтобы новый
  обработчик подключался без правок ядра.
- **Разнести ядро и фичи по отдельным пакетам.** Выделить core (`WaitGroup`, `State`,
  `FeatureExecutor`, транспорт, обёртка расширения) в самостоятельный пакет, а фичи
  (MongoDB, Sleeper, будущие) — в отдельные пакеты-плагины поверх него. Это уберёт у ядра
  зависимость от конкретных фич (например, `ext-mongodb` нужен только пакету MongoDB),
  позволит версионировать и подключать фичи независимо и упростит добавление новых.
- **Кросс-флоу конкурентность (вложенные корутины).** Сейчас, если внутри корутины
  запущено несколько корутин (вложенный `WaitGroup`), они выполняются **последовательно**
  относительно внешнего флоу. Нужно, чтобы вложенные корутины тоже работали **конкурентно** —
  и между собой, и параллельно с задачами внешнего флоу, а не блокировали его до своего
  завершения.
- **Остановка корутины из любой точки выполнения.** Возможность отменить отдельную корутину
  (а не только весь флоу) из произвольного места — как изнутри самой корутины, так и снаружи,
  с корректной отменой связанной Go-задачи и освобождением ресурсов.
