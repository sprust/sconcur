# MySQL (универсальная SQL-фича)

Асинхронная работа с реляционной БД поверх Go `database/sql`. Каждый запрос
уходит в Go-расширение и выполняется в горутине, пока корутина приостановлена, —
десятки запросов летят «веером». Вне `WaitGroup` тот же API работает синхронно.

Фича спроектирована универсально: ядро (`SConcur\Features\Sql`) драйвер-агностично,
а `SConcur\Features\Mysql\Connection` — тонкий фасад, задающий драйвер MySQL.
PgSQL добавляется зеркальным фасадом без изменения ядра.

## Быстрый старт

```php
$connection = new \SConcur\Features\Mysql\Connection(
    dsn: 'user:pass@tcp(127.0.0.1:3306)/app?parseTime=true',
    timeoutMs: 5000,
);

// стриминг строк (большие выборки не буферизуются целиком)
foreach ($connection->query('SELECT id, name FROM users WHERE age > ?', [18]) as $row) {
    echo $row['name'] . PHP_EOL;
}

// вся выборка сразу
$rows = $connection->fetchAll('SELECT * FROM users WHERE id = ?', [$id]);

// изменение: affectedRows + lastInsertId
$result = $connection->exec('INSERT INTO users (name) VALUES (?)', ['Ann']);
echo $result->lastInsertId;
```

В `WaitGroup::add(...)` те же вызовы исполняются конкурентно.

## DSN и биндинги

- **DSN** — формат драйвера go-sql-driver/mysql:
  `user:pass@tcp(host:port)/dbname?param=value`. Полезные параметры:
  `parseTime=true` (даты как `time.Time`), `charset`, `loc`.
- **`interpolateParams=true` включён по умолчанию** фасадом `Mysql\Connection`:
  запрос с биндингами уходит одним round-trip'ом (COM_QUERY с клиентской
  интерполяцией значений), а не PREPARE + EXECUTE + CLOSE. Это заметно быстрее на
  синхронном пути (≈−50% на exec, ≈−75% на select по замерам) и совпадает с
  поведением PDO по умолчанию; экранирование (защита от инъекций) выполняет драйвер.
  Чтобы вернуть серверные prepared-выражения, передайте `interpolateParams=false`
  в DSN явно.
- **Плейсхолдеры** — нативные для драйвера: у MySQL это `?`. Биндинги — всегда
  позиционный список, передаётся в драйвер как есть (без переписывания), что
  защищает от SQL-инъекций.

## Транзакции

Транзакция закреплена за одним соединением на серии команд. Открывается через
`begin()`; завершается `commit()` или `rollback()`.

```php
$transaction = $connection->begin();

try {
    $transaction->exec('UPDATE accounts SET balance = balance - ? WHERE id = ?', [100, $from]);
    $transaction->exec('UPDATE accounts SET balance = balance + ? WHERE id = ?', [100, $to]);
    $transaction->commit();
} catch (\Throwable $exception) {
    $transaction->rollback();

    throw $exception;
}
```

`begin(int $isolationLevel = 0, bool $readOnly = false)` принимает уровень
изоляции (значения `sql.IsolationLevel`; `0` — уровень по умолчанию драйвера) и
флаг read-only. У объекта `Transaction` те же `query`/`fetchAll`/`exec`, что и у
соединения.

**Откат при обрыве.** Если транзакция брошена без `commit()/rollback()`
(исключение, ранний выход, остановка `WaitGroup`), Go-сторона откатывает её
автоматически: транзакция привязана к контексту, который отменяется при остановке
флоу, и `database/sql` делает rollback.

**Конкурентные транзакции.** В одном `WaitGroup` каждая корутина держит свою
транзакцию (своё соединение из пула) — транзакции выполняются параллельно.

## Параметры соединения

Конструктор `Connection`:

| Параметр            | По умолчанию | Назначение |
|---------------------|--------------|------------|
| `dsn`               | —            | строка подключения драйвера |
| `timeoutMs`         | 30000        | предельное время одной операции query/exec |
| `maxOpenConns`      | без лимита   | `SetMaxOpenConns` пула |
| `maxIdleConns`      | по умолч. Go | `SetMaxIdleConns` пула |
| `connMaxLifetimeMs` | без лимита   | `SetConnMaxLifetime` пула |

`timeoutMs` ограничивает каждый отдельный query/exec (обязательное предельное
время выполнения). На время жизни самой транзакции он не распространяется —
транзакция живёт до commit/rollback или остановки флоу.

## Пул соединений и конкурентность

Каждая операция выполняется на соединении из пула `*sql.DB`, который живёт в
Go-расширении и переиспользуется между задачами и корутинами. Пул общий для всех
`Connection` с одинаковыми DSN и размерами пула (ключ — `драйвер+dsn+размеры`);
сам `*sql.DB` потокобезопасен. Незанятый пул, не использованный дольше 5 минут,
закрывается; все пулы закрываются при остановке расширения.

Размер пула задаётся в конструкторе `Connection`:

- `maxOpenConns` — максимум одновременных соединений (`0` — без лимита);
- `maxIdleConns` — сколько простаивающих держать (по умолчанию — дефолт Go, 2);
- `connMaxLifetimeMs` — максимальное время жизни соединения.

**Важно про веер.** В `WaitGroup` каждая autocommit-операция и каждая транзакция
берёт **отдельное** соединение на время своего выполнения. Запустив `N` корутин с
неограниченным пулом, вы откроете до `N` соединений одновременно — и упрётесь в
лимит сервера (`max_connections`, ошибка MySQL `1040 Too many connections`).

Ограничьте пул через `maxOpenConns`:

```php
$connection = new \SConcur\Features\Mysql\Connection(
    dsn: 'user:pass@tcp(127.0.0.1:3306)/app',
    maxOpenConns: 50,
);
```

- **Autocommit-запросы.** При насыщенном пуле `database/sql` ставит вызовы в
  очередь и ждёт освобождения соединения (backpressure), а не падает с ошибкой.
- **Транзакции.** Каждая транзакция удерживает соединение на всю свою жизнь
  (begin → … → commit/rollback). При `maxOpenConns` меньше числа корутин лишние
  `begin()` блокируются, пока не освободятся другие, — транзакции проходят волнами.
  Поэтому пул держите не меньше ожидаемого числа **одновременных** транзакций, либо
  ограничивайте сам веер (см. планируемый `WaitGroup::create(maxConcurrency: N)`).

Подбирайте `maxOpenConns` так, чтобы он был `<= max_connections` сервера и
соответствовал желаемой степени параллелизма.

## Внутреннее устройство

- **Пул соединений** (`ext/internal/features/sql/pools.go`) — реестр `*sql.DB`
  по ключу `драйвер+dsn+размеры`, с refcount и вытеснением простаивающих пулов
  (как пул клиентов MongoDB). `*sql.DB` потокобезопасен и переиспользуется между
  задачами.
- **Стриминг SELECT** (`rows_state.go`) — состояние держит `*sql.Rows`, отдаёт
  строки батчами (`batchSize`, по умолчанию 50) с просмотром на одну строку вперёд,
  чтобы определить наличие следующего батча. Закрывается при исчерпании, раннем
  `break` или остановке флоу.
- **Транзакция** (`transactions.go`) — `begin` кладёт `*sql.Tx` в `sync.Map` по
  id (ключ держащей задачи) и оставляет задачу «живой» (`hasNext`), чтобы соединение
  переживало серию команд. `commit`/`rollback` финализируют транзакцию и освобождают
  держащую задачу; финализация идемпотентна.

## Ограничения

- **Типы значений.** Значения возвращаются типизированными независимо от наличия
  биндингов: целые — `int`, `FLOAT`/`DOUBLE` — `float`, `VARCHAR`/`TEXT`/`CHAR` и
  бинарные (`BLOB`/`BINARY`) — строка, `DECIMAL` — строка (чтобы не терять
  точность), `DATE`/`DATETIME`/`TIMESTAMP` при `parseTime=true` в DSN — строка
  RFC3339, `NULL` — `null`. Беззнаковый `BIGINT` больше `PHP_INT_MAX` выходит за
  диапазон знакового 64-битного `int` — храните/читайте такие значения как строку.
- **Курсор внутри транзакции** нужно либо дочитать до конца, либо использовать
  `fetchAll`, прежде чем выполнять следующую команду той же транзакции, — иначе
  соединение занято открытым курсором.
- Применимы общие ограничения библиотеки: только CLI/NTS, никаких `pcntl_fork`
  после загрузки расширения, не завершать процесс при активных задачах
  (см. [README](../README.md)).
