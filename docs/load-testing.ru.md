# Нагрузочное тестирование: поведение сервера под нагрузкой

Этот документ описывает, как нагрузить HTTP-сервер набором async-I/O-фич сразу и снять
при этом потребление памяти и CPU, а также фиксирует результаты эталонного прогона и
выводы из них.

Инструменты:
- ручка `/all` демо-сервера (`tests/servers/http/http-server.php`) — на каждый запрос
  конкурентно (вложенный `WaitGroup`) гоняет бэкенд-I/O-фичи: `Sleeper`, MongoDB
  (insert + findOne), MySQL (`INSERT` + `SELECT 1`), PostgreSQL (`INSERT` + `SELECT 1`).
  HTTP-клиент исключён намеренно: его self-hit в свой же `/` заставлял бы каждый `/all`
  обслуживать второй запрос и искажал бы rps (он покрыт своими бенчами). Соединения
  создаются лениво, один раз на воркер (Go-сторона пулит их по URI/DSN), поэтому
  остальные демо-роуты их не платят и не требуют поднятых БД. Зафиксированные ниже
  результаты сняты, когда SQL-часть была ещё без `INSERT` (только `SELECT 1`);
- скрипт `tests/benchmarks/http-load-stats.sh` (`make bench-http-load-stats`) — поднимает
  пул серверов (`SO_REUSEPORT`, по процессу на ядро), гонит `wrk` по `/all` и во время
  прогона семплит `docker stats` (CPU%/MEM по контейнерам сервера и БД) плюс суммарный
  RSS воркеров из `/proc/<pid>/status` (детект утечки);
- скрипт `tests/benchmarks/rr-load-stats.sh` (`make bench-rr-load-stats`) — тот же
  гарнесс, но сервер — RoadRunner-эталон на нативных драйверах
  (`tests/servers/roadrunner/`): один `rr serve` со своим пулом воркеров, те же
  правила pinning'а и семплинга, цифры сравнимы с `http-load-stats.sh` напрямую.

## Как запустить

Нужны `wrk` на хосте и поднятые сервисы (`make up`: `php`, `mongodb`, `mysql`,
`postgres`). Запускать с хоста (там живут `wrk` и `docker`), серверы крутятся в
контейнере (там собрано расширение).

```sh
make bench-http-load-stats
# тюнинг через env:
SERVERS=12 WRK_THREADS=4 CONNECTIONS=256 DURATION=20 SAMPLE_INTERVAL=2 \
    tests/benchmarks/http-load-stats.sh

# базовый прогон по пустой ручке "/" (потолок самого фреймворка, без I/O):
make bench-http-load-stats-empty

# soak-режим: длинный устойчивый прогон с трендом RSS во времени и наклоном
# (MiB/мин) — для детекта МЕДЛЕННОЙ утечки. Дефолты: 10 минут, семпл 15 c.
make bench-http-load-soak               # 10 минут по умолчанию
DURATION=3600 make bench-http-load-soak # часовой soak
```

`MODE=soak` дополнительно печатает таблицу `тренд (elapsed → RSS)` и наклон
least-squares в MiB/мин с вердиктом: `стабильно` / `рост — возможна утечка` /
`снижается (GC/возврат памяти)`.

Методология как у [`http-throughput.sh`](../tests/benchmarks/http-throughput.sh): серверы
и генератор нагрузки прибиты к непересекающимся ядрам (`taskset`), а `wrk` бьёт по
bridge-IP контейнера напрямую, минуя docker-proxy (NAT режет throughput).

## Базовый прогон (пустая ручка)

Окружение: ноутбук Intel i7-13620H (6 P-ядер/12 HT + 4 E-ядра = 16 потоков), сервисы в
Docker, 12 серверов / 4 wrk-ядра, 256 соединений, 20 c. `wrk` бьёт по `/` (ответ `ok`,
без I/O и без заметного CPU) — это потолок связки HTTP-сервер + фреймворк, пол, на
который сверху ложится налог фич в `/all`. Запуск: `make bench-http-load-stats-empty`.

| Метрика | `/` (пусто) | `/all` (все фичи) |
|---|---|---|
| Throughput | **≈25 500 req/sec** | 2 910 req/sec |
| Латентность | p50 10 ms · p90 14 ms · p99 20 ms | p50 81 · p90 119 · p99 196 ms |
| CPU серверов (`php`) | avg ~1180 % | avg 976 % |
| RSS воркеров (сумма 12) | ~552 MiB (плоско) | ~610 MiB |

(Две прогонки дали 25.4k и 25.6k req/sec, 0 ошибок — все ответы `200`.) Разрыв ~9× —
это цена фан-аута по фичам: на `/all` каждый запрос делает 4-кратный конкурентный fan-out
через границу PHP↔Go (msgpack + спавн/планирование фибр), и throughput упирается именно
в эти накладные расходы, а не в БД. На пустой ручке их нет — остаётся чистый потолок
сервера (CPU при этом даже выше: запросов в секунду кратно больше). Детальный прогон
`/all` — [ниже](#эталонный-прогон).

## Эталонный прогон

Та же машина и параметры (12 серверов / 4 wrk-ядра, 256 соединений, 20 c), но `wrk` бьёт
по `/all`. Запросы к БД тривиальные (`SELECT 1`, `findOne`) — меряется потолок самого
фреймворка, а не БД. `/all` гоняет все async-I/O-фичи разом — `Sleeper`, MongoDB
(insert + findOne), MySQL (`SELECT 1`), PostgreSQL (`SELECT 1`), — то есть 2 910 req/sec
ниже это полный 4-кратный фан-аут на каждый запрос, а не пустая ручка (голый потолок
сервера — в [базовом прогоне](#базовый-прогон-пустая-ручка) выше, ≈25.5k req/sec).

| Метрика | Значение |
|---|---|
| Throughput | **2 910 req/sec** (0 ошибок — все 4 фичи `ok`) |
| Латентность | p50 81 ms · p90 119 ms · p99 196 ms |
| **RSS воркеров (сумма 12 шт.)** | first **608.9** / peak **610.6** / last **610.6 MiB** → дрейф **+1.6 MiB** |
| CPU серверов (`php`) | avg **976 %** / peak 1053 % (≈ 10 из 12 ядер) |
| CPU бэкендов | PostgreSQL 141 %/234peak · MongoDB 112 %/141peak · MySQL 27 % |
| MEM (контейнеры) | php 326 · mongo 472 · mysql 143 · pg 106 MiB |

(Дрейф RSS за 20 c — это шум прогрева; авторитетный вердикт по утечке даёт soak ниже.)

## Soak-прогон (10 минут)

То же окружение, устойчивая нагрузка 128 соединений в течение 600 c (детект медленной
утечки). За прогон — 2 058 645 запросов (~8.2M feature-операций).

| Метрика | Значение |
|---|---|
| Throughput | **3 431 req/sec** · p50 37 ms · p99 61 ms |
| RSS воркеров (тренд за 10 мин) | first **606.6** / peak **606.6** / last **603.1 MiB** |
| **Дрейф / наклон** | **−3.5 MiB** / **−0.08 MiB/мин** → вердикт **стабильно** |
| CPU серверов (`php`) | avg 1120 % / peak 1179 % |

RSS держался плоско (602–607 MiB) всю дистанцию — наклон −0.08 MiB/мин в пределах шума
(короткий 20-c прогон выше дал ложный «рост» из-за прогрева — soak это снимает).
**Медленной утечки нет.** MEM контейнера `mongodb` при этом вырос до ~596 MiB — из-за
безлимитных `insert` ручки `/all` в коллекцию `load_all`; RSS воркеров SConcur не
дрогнул. Данные накапливает БД, а сторона SConcur стабильна (коллекцию `load_all` после
прогонов можно дропнуть).

## Выводы

1. **Память стабильна — главный результат.** ~50 MiB RSS на воркер, а 10-минутный soak
   (2.06M запросов) держал RSS плоско с наклоном −0.08 MiB/мин (= шум) — медленной утечки
   нет. Для долгоживущего сервера это ключевой сигнал: связка Go-рантайм + PHP-фибры +
   пулы соединений + граница PHP↔Go ничего не накапливает. Согласуется с `MemLeakTest`.

2. Устойчивость. Насыщение с 4-кратным конкурентным фан-аутом на запрос → 0 ошибок,
   p99 < 200 ms. Кооперативный планировщик и вложенные `WaitGroup` держат высокую
   конкурентность.

3. Узкое место — CPU самого SConcur, не БД. ~10 ядер на серверы против ~1–2 на каждую БД.
   Стоимость запроса — это накладные расходы фреймворка (msgpack, спавн/планирование
   фибр, 4× переход PHP↔Go), а не сами `SELECT 1`/`findOne`. На дешёвых бэкенд-операциях
   доминирует налог фреймворка (отсюда ~2.9k rps), а на реальной нагрузке (медленные
   запросы, сетевые задержки) этот налог амортизируется и раскрывается сильная сторона
   SConcur — конкурентность I/O веером.

## Про «CPU в небесах»

Высокий CPU под нагрузкой — это насыщение, а не аномалия. `wrk` гонит сервер на максимум,
поэтому ~1000 % (≈ все 12 ядер по 100 %) означает «нашли потолок», а не баг или утечку.
Значимая метрика — не сам CPU %, а throughput и CPU на запрос: под сатурацией CPU % всегда
у потолка, различия видны в throughput. В продакшене сервер крутится ниже сатурации, и
CPU пропорционален нагрузке, а не прибит к 100 %.

`/all` тяжёлый по дизайну: 4-кратный фан-аут, каждая фича — round-trip PHP↔Go (msgpack +
спавн/планирование фибры). На дешёвых бэкенд-операциях это и есть потолок (~2.9k rps), на
реальной (медленной) нагрузке — амортизируется. HTTP-клиент из `/all` исключён намеренно
(см. вступление): его self-hit удваивал бы обслуживаемую нагрузку и искажал бы rps (раньше
из-за него `/all` показывал лишь ~1.7k rps).

## Оговорки

- Синтетика на ноутбуке. Потребительский CPU (P+E-ядра, HyperThreading, all-core
  throttling) занижает масштабирование по ядрам; на серверном CPU цифры были бы выше и
  линейнее.
- Тривиальные запросы недооценивают суть SConcur (конкурентный I/O). Для честного
  позиционирования I/O-bound сценарий — отдельные бенчи `bench-http-server-io`.
- Дистанция. 10-минутный soak (2.06M запросов) уже подтвердил отсутствие медленной утечки
  (наклон −0.08 MiB/мин). Для абсолютной уверенности на проде — многочасовой прогон:
  `make bench-http-load-soak` (`MODE=soak`), напр. `DURATION=3600` и дольше.

## WebSocket-сервер под нагрузкой

У WebSocket-сервера есть та же связка нагрузка + ресурсы, но `wrk` тут не годится (он
только HTTP), поэтому генератор нагрузки — свой: `ext/cmd/ws-load` (Go, на
`coder/websocket`), WS-аналог `wrk`. Он держит N постоянных соединений, гоняет по ним
back-to-back round-trip'ы и печатает throughput и латентность p50/p90/p99.

Инструменты:
- команда `all` демо-сервера (`tests/servers/ws/ws-server.php`) — на каждое сообщение
  конкурентно (вложенный `WaitGroup`) гоняет те же бэкенд-I/O-фичи, что и HTTP-`/all`:
  `Sleeper`, MongoDB (insert + findOne), MySQL (`INSERT` + `SELECT 1`), PostgreSQL (`INSERT` + `SELECT 1`).
  Соединения к БД создаются лениво, один раз на воркер;
- скрипт `tests/benchmarks/ws-load-stats.sh` (`make bench-ws-load-stats`) — поднимает пул
  ws-серверов (`SO_REUSEPORT`, по процессу на ядро) в контейнере `php`, гонит по нему
  `ext/cmd/ws-load` и во время прогона семплит `docker stats` плюс суммарный RSS воркеров
  (детект утечки), той же логикой, что и `http-load-stats.sh`.

Отличие от HTTP-варианта: и пул, и генератор живут в контейнере `php` (WebSocket не
требует хостовых утилит), пиннятся на непересекающиеся ядра (`taskset`), генератор бьёт по
пулу через loopback (`127.0.0.1`, без NAT).

```sh
make bench-ws-load-stats
# тюнинг через env:
SERVERS=12 CONNECTIONS=256 DURATION=20 SAMPLE_INTERVAL=2 tests/benchmarks/ws-load-stats.sh

# базовый прогон по "ping" (потолок самого фреймворка, без I/O):
make bench-ws-load-stats-empty

# soak-режим (длинный устойчивый прогон с трендом RSS и наклоном MiB/мин):
make bench-ws-load-soak               # 10 минут по умолчанию
DURATION=3600 make bench-ws-load-soak # часовой soak
```

Метрики читаются так же, как у HTTP: `ping` против `all` показывает цену
фичевого фан-аута (round-trip PHP↔Go на каждую фичу), а наклон RSS в soak-режиме —
авторитетный вердикт по утечке (короткий прогон даёт шум прогрева). Стороны WS-сервера и
HTTP-сервера здесь устроены одинаково, поэтому и выводы переносятся.
