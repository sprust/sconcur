# Нагрузочное тестирование: поведение сервера под нагрузкой

Этот документ описывает, как нагрузить HTTP-сервер **всеми** async-I/O-фичами сразу и
снять при этом потребление памяти и CPU, а также фиксирует результаты эталонного
прогона и выводы из них.

Инструменты:
- ручка **`/all`** демо-сервера (`tests/servers/http/http-server.php`) — на каждый
  запрос конкурентно (вложенный `WaitGroup`) гоняет **каждую** фичу из `MethodEnum`:
  `Sleeper`, MongoDB (insert + findOne), MySQL (`SELECT 1`), PostgreSQL (`SELECT 1`) и
  HTTP-клиент (self-hit своего же `/`). Соединения создаются **лениво, один раз на
  воркер** (Go-сторона пулит их по URI/DSN), поэтому остальные демо-роуты их не платят
  и не требуют поднятых БД;
- скрипт **`tests/benchmarks/http-load-stats.sh`** (`make bench-http-load-stats`) —
  поднимает пул серверов (`SO_REUSEPORT`, по процессу на ядро), гонит `wrk` по `/all` и
  во время прогона семплит `docker stats` (CPU%/MEM по контейнерам сервера и БД) плюс
  суммарный RSS воркеров из `/proc/<pid>/status` (детект утечки).

## Как запустить

Нужны `wrk` на хосте и поднятые сервисы (`make up`: `php`, `mongodb`, `mysql`,
`postgres`). Запускать **с хоста** (там живут `wrk` и `docker`), серверы крутятся в
контейнере (там собрано расширение).

```sh
make bench-http-load-stats
# тюнинг через env:
SERVERS=12 WRK_THREADS=4 CONNECTIONS=256 DURATION=20 SAMPLE_INTERVAL=2 \
    tests/benchmarks/http-load-stats.sh

# soak-режим: длинный устойчивый прогон с трендом RSS во времени и наклоном
# (MiB/мин) — для детекта МЕДЛЕННОЙ утечки. Дефолты: 10 минут, семпл 15 c.
make bench-http-load-soak               # 10 минут по умолчанию
DURATION=3600 make bench-http-load-soak # часовой soak
```

`MODE=soak` дополнительно печатает таблицу `тренд (elapsed → RSS)` и **наклон**
least-squares в MiB/мин с вердиктом: `стабильно` / `рост — возможна утечка` /
`снижается (GC/возврат памяти)`.

Методология честности — как у [`http-throughput.sh`](../tests/benchmarks/http-throughput.sh):
серверы и генератор нагрузки прибиты к **непересекающимся** ядрам (`taskset`), а `wrk`
бьёт по **bridge-IP** контейнера напрямую, минуя docker-proxy (NAT, который тихо режет
throughput).

## Эталонный прогон

**Окружение:** ноутбук Intel i7-13620H (6 P-ядер/12 HT + 4 E-ядра = 16 потоков),
сервисы в Docker, 12 серверов / 4 wrk-ядра, 256 соединений, 20 c. **Запросы к БД
тривиальные** (`SELECT 1`, `findOne`) — то есть меряется потолок самого фреймворка, а
не БД.

| Метрика | Значение |
|---|---|
| Throughput | **1 668 req/sec** (0 ошибок — все 5 фич `ok`) |
| Латентность | p50 142 ms · p90 219 ms · p99 395 ms |
| **RSS воркеров (сумма 12 шт.)** | first **624.8** / peak **624.8** / last **623.7 MiB** → дрейф **−1.1 MiB** |
| CPU серверов (`php`) | avg **939 %** / peak 1032 % (≈ 9–10 из 12 ядер) |
| CPU бэкендов | PostgreSQL 91 %/204peak · MongoDB 85 %/115peak · MySQL 25 % |
| MEM (контейнеры) | php 364 · mongo 233 · mysql 158 · pg 94 MiB |

## Soak-прогон (10 минут)

То же окружение, устойчивая нагрузка 128 соединений в течение **600 c** (детект
**медленной** утечки). За прогон — **1 552 494 запроса** (~7.76M feature-операций).

| Метрика | Значение |
|---|---|
| Throughput | **2 587 req/sec** · p50 48 ms · p99 89 ms |
| RSS воркеров (тренд за 10 мин) | first **611.0** / peak **612.7** / last **612.2 MiB** |
| **Дрейф / наклон** | **+1.2 MiB** / **+0.10 MiB/мин** → вердикт **стабильно** |
| CPU серверов (`php`) | avg 1147 % / peak 1217 % |

RSS держался **плоско** (611–612.7 MiB) всю дистанцию — наклон +0.10 MiB/мин в пределах
шума. **Медленной утечки нет.** При этом MEM контейнера `mongodb` вырос до ~610 MiB —
из-за безлимитных `insert` ручки `/all` в коллекцию `load_all`; RSS воркеров SConcur при
этом **не дрогнул**. То есть данные накапливает БД, а **сторона SConcur стабильна**
(коллекцию `load_all` после прогонов можно дропнуть).

## Выводы

1. **Память стабильна — главный результат.** На коротком прогоне RSS не вырос (~52 MiB
   на воркер), а **10-минутный soak** (1.55M запросов) держал RSS **плоско** с наклоном
   **+0.10 MiB/мин** (= шум) — **медленной утечки нет**. Для долгоживущего сервера это
   ключевой сигнал: связка Go-рантайм + PHP-фибры + пулы соединений + граница PHP↔Go
   **ничего не накапливает**. Согласуется с `MemLeakTest`.

2. **Устойчивость.** Насыщение с 5-кратным конкурентным фан-аутом на запрос → **0
   ошибок**, p99 < 400 ms. Кооперативный планировщик и вложенные `WaitGroup` держат
   высокую конкурентность.

3. **Узкое место — CPU самого SConcur, не БД.** ~10 ядер на серверы против ~1–2 на
   каждую БД. Стоимость запроса — это **накладные расходы фреймворка** (msgpack,
   спавн/планирование фибр, 5× переход PHP↔Go), а не сами `SELECT 1`/`findOne`. Отсюда:
   на **дешёвых** бэкенд-операциях доминирует «налог» фреймворка (отсюда скромные
   ~1.7k rps), а на **реальной** нагрузке (медленные запросы, сетевые задержки) этот
   налог амортизируется и раскрывается сильная сторона SConcur — конкурентность I/O
   «веером».

## Оговорки

- **Синтетика на ноутбуке.** Потребительский CPU (P+E-ядра, HyperThreading,
  all-core throttling) занижает масштабирование по ядрам; на серверном CPU цифры были
  бы выше и линейнее.
- **Тривиальные запросы** недооценивают суть SConcur (конкурентный I/O). Для честного
  позиционирования I/O-bound сценарий — отдельные бенчи `bench-http-reuseport-io`.
- **Дистанция.** 10-минутный soak (1.55M запросов) уже подтвердил отсутствие медленной
  утечки (наклон +0.10 MiB/мин). Для абсолютной уверенности на проде — многочасовой
  прогон: `make bench-http-load-soak` (`MODE=soak`), напр. `DURATION=3600` и дольше.
