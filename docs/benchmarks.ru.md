# Бенчмарки фич

Замеры производительности по каждой фиче (кроме `Sleeper`): что стоит вызов через
границу PHP↔Go по сравнению с родным драйвером и что даёт конкурентный запуск веером.
Цифры — ориентир, а не гарантия: они сняты на конкретной машине и зависят от железа,
настроек БД и нагрузки.

> Колонка `sync` несёт фиксированный оверхед, не присущий подходу: синхронный вызов
> (вне `WaitGroup`) всё ещё идёт через планировщик и машинерию Fiber. Его убирают в планах
> (см. [README](../README.ru.md#планы)), после чего вызов вне корутины пойдёт в Go
> напрямую. До тех пор цифры `sync` непоказательны — значимо сравнение `native` и `async`.

## Коротко

- Веер (`async`) выигрывает там, где у операции есть реальная цена — fsync на
  запись, серверная работа по датасету, сетевое ожидание: SQL-записи ~3–18×, тяжёлые
  чтения (`count`) ~2–7.5×, клиенты с задержкой 100 мс ~44×, `/all` против
  RoadRunner ~6×.
- Дешёвые точечные операции остаются за родным драйвером (`selectOne`/`findOne`,
  `selectMany`, однодокументные операции MongoDB): граница PHP↔Go стоит дороже самой
  операции, а накладывать нечего.
- Одиночный вызов через SConcur всегда дороже нативного (конвертация на границе) —
  выгоду даёт только конкурентность.

## Окружение

- CPU: Intel Core i7-13620H (16 потоков), 15 GiB RAM, Linux.
- Всё в Docker (`docker-compose.yml`), бенчмарки запускались из контейнера `php`
  (`make bench-*`), серверные пулы — из контейнера `servers` (3 воркера,
  `SO_REUSEPORT`).
- Версии PHP/Go/драйверов/серверов БД — см. [«Версии, на которых
  тестировалось»](../README.ru.md#версии-на-которых-тестировалось) в README.

Бенчмарки снимаются с данными БД на диске хоста (SSD) — как в реальном
размещении: записи платят настоящий fsync, чтения по горячим страницам идут из
кэша БД (по задержке — почти память). В `docker-compose.yml` по умолчанию
данные БД лежат в `tmpfs` (быстрые тесты, состояние сбрасывается рестартом);
для бенч-сессии в файле раскомментируются именованные тома (и комментируются
`tmpfs`-блоки):

| БД | Каталог данных | Том |
| --- | --- | --- |
| MongoDB | `/data/db` | `mongodb-data` |
| MySQL | `/var/lib/mysql` | `mysql-data` |
| PostgreSQL | `/var/lib/postgresql/data` | `postgres-data` |

Перед бенч-сессией состояние БД сбрасывается командой `make bench-reset`
(удаляет тома и пересоздаёт контейнеры): без сброса записи накапливаются между
прогонами и цифры дрейфуют.

Клиентские и серверные цифры сняты 2026-07-22, цифры БД — 2026-07-23 (по
методологии холодного датасета ниже), на ненагруженной машине.

## Оверхед на конвертацию (граница PHP↔Go)

Каждый вызов фичи пересекает границу PHP↔Go, и данные на этой границе конвертируются:
аргументы упаковываются в MessagePack (msgpack-размеченные DTO,
`Transport/MessagePackTransport`), результат распаковывается обратно; для MongoDB
документы дополнительно проходят BSON-(де)сериализацию через `ext-mongodb`. Это
фиксированная CPU-цена на каждую операцию, поверх cgo-вызова и диспетчеризации
горутины.

На дешёвых чтениях из кэша БД этот оверхед виден напрямую как разрыв между `native` и
`sync` (оба идут по очереди, но `sync` — через Go): например, `pgsql-selectOne`
3.8 → 10.2 мс на 100 вызовов, `mysql-selectOne` 3.9 → 15.2 мс. Сам запрос почти
мгновенный, поэтому цена конвертации доминирует. На «медленной» операции (fsync,
сеть, тяжёлый запрос) та же фиксированная надбавка становится малой долей общего
времени, и разрыв в процентах сжимается.

Смысл SConcur не в ускорении одиночного вызова (тут родной драйвер всегда быстрее), а в
конкурентном запуске множества операций веером: одна фиксированная надбавка на вызов
окупается тем, что десятки ожиданий I/O накладываются, а не суммируются.

## Методология

Замерялись три режима на каждую фичу:

- `native` — эталон без SConcur: родной драйвер/механизм PHP (драйвер `mongodb/mongodb`,
  `PDO` для MySQL/PostgreSQL, `file_get_contents`/stream-обёртки для HTTP, сырые сокеты
  для TCP/WebSocket). Вызовы идут по очереди.
- `sync` — SConcur вне `WaitGroup` (синхронный путь `Extension::wait`): тот же путь
  через Go, но операции выполняются по очереди.
- `async` — SConcur внутри `WaitGroup`: N корутин запускаются веером, планировщик
  собирает результаты по мере готовности.

Каждый бенчмарк БД (MongoDB, MySQL, PostgreSQL) прогонялся 5 раз, каждый
клиентский/серверный — 3 раза; в таблицах — медиана. Каждый прогон БД стартует с
холодных данных: перед замером бенчмарк-таблица/коллекция дропается и заново сидится
до датасета в 100 000 строк/документов родным драйвером батчами (multi-row `INSERT` в
одной транзакции для SQL, id = 1..N, сиквенс PG сдвигается за них; чанками `insertMany`
для MongoDB с целочисленным `_id` 1..N и полями под фильтры бенчмарков). Прогоны
полностью независимы — между прогонами и между режимами ничего не накапливается.

Точечные операции работают по уникальным id: каждому режиму выделен свой диапазон id
внутри посеянного датасета, и каждый вызов (включая прогрев) бьёт в свою
строку/документ. Поэтому `selectOne`/`findOne`/`updateOne`/`update`/`deleteOne`/`delete`
не делят горячую строку, каждый delete реально удаляет строку, а лок общей строки не
может сериализовать веер. `selectMany` читает окно в 100 строк, скользящее по диапазону
режима; `insert`/`insertOne`/`insertMany`/`transaction` вставляют в таблицу, уже
наполненную датасетом; `count`/`updateMany`/`aggregate` работают по всему датасету
(посеянные документы подходят под их фильтры).

Число вызовов (`count`) на режим: 100 по умолчанию, 50 для клиентских I/O-бенчмарков
(каждый вызов — 100 мс сна на сервере). Три mongo-бенчмарка ограничены природой
операции: `createIndex` 20 (лимит MongoDB — 64 индекса на коллекцию, 3×20 = 60 за
прогон умещается), `bulkWrite` 20 (фильтры bulk-операций без индексов — каждый вызов
сканирует датасет несколько раз) и `updateMany` 10 (каждый вызов переписывает все
100 000 документов). Одиночные запуски — `make bench-<name> c=<count>`; вся БД-сессия
(5 холодных прогонов на бенчмарк плюс агрегация median/min/max) —
`make bench-db-runs` (`tests/benchmarks/db-bench-runs.sh`).

Перед замером — прогрев (отбрасывается): по несколько последовательных вызовов
native/sync и один async-веер полного размера. Это выравнивает условия: native входит
в замер с установленным соединением и prepared statement, а без прогрева async-веер
платил бы за раскрутку пула соединений (Go-пул и драйвер Mongo открывают соединения по
требованию до ширины веера) прямо внутри замеряемой фазы. У `createIndex` прогрева нет
(лимит индексов). SQL-пулы бенчей: `maxOpenConns: 50`; `maxIdleConns` по умолчанию
равен `maxOpenConns`, чтобы пул не схлопывался между веерами.

Память — пик RSS PHP-процесса (`memory_get_peak_usage`) за режим, а не на вызов.

В таблицах есть сравнение `async к native` — знаковый процент
`(native − async) / native`, с ✅, когда веер (`async`) быстрее родного драйвера, и ❌,
когда медленнее. В таблицах клиентов это отдельная колонка; в таблицах БД процент стоит
в скобках прямо в той ячейке, к которой относится, — у колонок медианы, min и max свой
(по медианам и по min- и max-значениям режимов за прогоны), что показывает разброс
сравнения между прогонами. В сравнении с RoadRunner колонка `к RoadRunner` — это
`(SConcur − RoadRunner) / RoadRunner` по throughput (✅ = у SConcur больше). Строки до
~50 мс чувствительны к шуму: их знак может меняться между прогонами — смена знака между
значениями `min` и `max` одной строки как раз это и показывает.

async обгоняет native там, где вызов заставляет сервер прожёвывать датасет: `count`
(327 мс vs 2282, ~7×; каждый вызов считает все 100 тыс. документов), `updateMany`
(317 vs 1741, ~5.5×; каждый вызов переписывает все 100 тыс.), `bulkWrite` (536 vs 3458,
~6.5×; фильтры bulk-операций без индексов сканируют коллекцию по несколько раз за
вызов) и `createIndex` (1620 vs 2194; каждый индекс строится по 100 тыс. документов) —
тяжёлая серверная работа накладывается по пулу соединений и ядрам сервера, вместо
очереди вызов за вызовом. Точечные однодокументные операции остаются за native —
`insertOne`, `updateOne`, `deleteOne`, `findOne` (каждый вызов бьёт в свой `_id`),
`insertMany`, `aggregate` (`$limit 30` рано обрывает скан) и no-op `command`: MongoDB
не платит fsync на операцию при дефолтном write concern, поэтому даже запись — быстрая
in-memory операция без ожидания I/O для наложения, и конвертация на границе стоит
дороже самой операции.

Диск переворачивает SQL-картину на записях: каждая зафиксированная запись платит
fsync, последовательные режимы суммируют его по всем 100 вызовам, а веер накладывает.
async быстрее native на mysql `insert` в ~14× (45.8 vs 642 мс), `update` в ~15×
(41.1 vs 624), `delete` в ~16× (40.5 vs 637), `transaction` в ~10× (69.1 vs 666); на
pgsql `insert` в ~5.5× (23.9 vs 132), `update` в ~18× (7.0 vs 126), `delete` в ~6×
(22.8 vs 132), `transaction` в ~3× (47.5 vs 151). Когда каждый вызов бьёт в свою
строку, `update` и `delete` ведут себя ровно как `insert` — артефакты общей строки из
старой методологии (лок, сериализующий веер, и no-op-удаления уже удалённого id)
устранены самой конструкцией. `count` по таблице в 100 000 строк тоже уходит к async:
mysql 76.2 vs 147 (~2×), pgsql 41.6 vs 314 (~7.5×; `COUNT(*)` в PostgreSQL сканирует
heap) — чтение, заставляющее сервер реально поработать, тоже стоит запускать веером.
За native остаются дешёвые точечные чтения из кэша (`selectOne`: pgsql 6.0 vs 3.8,
mysql 4.3 vs 3.9 — почти вровень) и `selectMany` (100 строк на вызов — доминирует
конвертация набора строк на границе: mysql 54.8 vs 8.5, pgsql 47.3 vs 8.8).

## Клиенты (HTTP / Socket / WebSocket)

Здесь виден смысл конкурентного режима. `native` и `sync` бьют по эндпоинту с задержкой
последовательно, `async` — веером. Эндпоинт `msleep` держит соединение 100 мс; 50
вызовов по очереди ≈ 5 c, а веером ≈ одному вызову.

| Бенчмарк | count | native, мс | sync, мс | async, мс | Память n/s/a, МБ | async к native |
| --- | ---: | ---: | ---: | ---: | --- | :---: |
| http-client (`/msleep/100`) | 50 | 5243 | 5222 | 120 | 4 / 4 / 4 | +98% ✅ |
| http-client-download (4 МиБ) | 50 | 1105 | 844 | 192 | 4 / 4 / 4 | +83% ✅ |
| socket-client (`msleep:100`) | 50 | 5222 | 5287 | 119 | 4 / 4 / 4 | +98% ✅ |
| ws-client (`msleep:100`) | 50 | 5255 | 5345 | 131 | 4 / 4 / 4 | +98% ✅ |

На I/O-задержке async даёт ~44× (5.2 c → 0.12 c): 50 ожиданий по 100 мс накладываются в
одно. `download` — скачивание тела 4 МиБ прямо в файл на стороне Go (не буферизуется в
PHP), поэтому память плоская, а веер всё равно ускоряет в ~6×.

| ws-throughput | 50 conn × 2000 × `ping` | 0.84 | ≈120 000 rt/s |

I/O-бенчмарки: 100 обработчиков, каждый спит 1 c асинхронно, — один кооперативный
процесс уже накладывает все ожидания, поэтому суммарное время ≈ одному сну (~1 c)
независимо от числа воркеров. CPU-бенчмарки: sha256-цикл не уступает управление, но пул
`SO_REUSEPORT` раскидывает 100 запросов по процессам/ядрам, поэтому 100 «неуступающих»
запросов проходят за ~0.7 c. Throughput меряет чистую цену round-trip под конкуренцией
(per-frame оверхед PHP↔Go плюс фрейминг): ~154 тыс. round-trip/с у socket и ~120 тыс. у
WebSocket на пуле. Поведение под длительной нагрузкой всеми фичами сразу — в
[docs/load-testing.ru.md](load-testing.ru.md).

### Пропускная способность HTTP: `/` vs `/all`

Отдельный замер сустейнутой пропускной способности под `wrk` (методология
[docs/load-testing.ru.md](load-testing.ru.md), скрипт `http-load-stats.sh`): скрипт сам
поднимает пул из `nproc − WRK_THREADS` = 12 процессов в контейнере `php` (ядра 0–11 за
серверами, wrk — на остальных), `wrk` бьёт по bridge-IP напрямую (минуя NAT). Параметры
прогона: 4 треда wrk, 256 соединений, 20 c, 3 прогона на ручку, все ответы `200`. Это не
тот же пул из 3 воркеров, что в таблице выше, — здесь потолок связки на всех серверных
ядрах.

- `/` — пустая ручка (ответ `ok`, без I/O и заметного CPU): чистый потолок HTTP-сервера
  и фреймворка.
- `/all` — на каждый запрос вложенный `WaitGroup` гоняет backend-I/O-фичи веером:
  MongoDB (insert + findOne), MySQL (`INSERT` + `SELECT 1`), PostgreSQL
  (`INSERT` + `SELECT 1`). SQL-пулы в ручке капнуты (`maxOpenConns: 5` на процесс):
  12–16 процессов с безлимитными пулами пробивают `max_connections = 100` у
  PostgreSQL, и часть запросов отвечала бы 500 «too many clients».

| Ручка | Requests/sec | Латентность p50 / p90 / p99 | CPU `php` avg / peak | MEM peak |
| --- | ---: | --- | --- | ---: |
| `/` (пусто) | ≈67 100 | 3.7 / 6.3 / 8.8 мс | ~1210% / ~1210% | ~256 MiB |
| `/all` (все фичи) | ≈2 680 | 87 / 165 / 267 мс | ~740% / ~765% | ~287 MiB |

Медиана из 3 прогонов; CPU `php` в процентах, потолок пула — 12 закреплённых ядер
(~1200%). На пустой ручке потолок упирается в CPU; `/all` на дисковых бэкендах
упирается уже не в CPU (~740% из 1200), а в fsync: каждый запрос — веер из 3 фич
(6 БД-операций, из них 3 записи) при пуле, капнутом 5 соединениями на процесс.

### Сравнение с RoadRunner (нативные драйверы)

Чтобы понять цену подхода, те же две ручки замерены на [RoadRunner](https://roadrunner.dev)
2025.1.15 — зрелом Go-сервере приложений для PHP. Условия идентичны: тот же контейнер
`php` (PHP 8.4.15 NTS), 12 воркеров, `wrk` 4 треда / 256 соединений / 20 c, 3 прогона,
удар по bridge-IP. Ручка `/all` RoadRunner-воркера намеренно не использует SConcur —
тот же функционал собран на нативных драйверах:

- `/` — PSR-7 воркер отдаёт `200 "ok"`.
- `/all` — тот же набор операций, что у SConcur, но нативно и последовательно (в
  воркере нет внутренней конкурентности): MongoDB `insertOne`+`findOne`
  (`mongodb/mongodb`), MySQL `INSERT`+`SELECT 1` (`PDO`), PostgreSQL `INSERT`+`SELECT 1`
  (`PDO`), ответ — та же JSON-мапа статусов.
- `/all-sconcur` — те же три фичи, но через SConcur веером во вложенном `WaitGroup`
  внутри того же RoadRunner-воркера (воркер загружает расширение) — сценарий
  «SConcur внутри RoadRunner-воркера»; запуск — `make bench-rr-sconcur-load-stats`.

Референсный RoadRunner-сервер закоммичен в `tests/servers/roadrunner/` (конфиг,
PSR-7 воркер с нативными копиями обеих ручек); бинарь `rr` ставится при сборке
контейнера `php`, запуск — `make rr-serve`. Нагрузочный замер по нему —
`tests/benchmarks/rr-load-stats.sh` (`make bench-rr-load-stats`, для `/` —
`make bench-rr-load-stats-empty`): тот же гарнесс, что `http-load-stats.sh`
(pinning ядер, wrk по bridge-IP, семплинг CPU/памяти и RSS воркеров), поэтому
цифры сравнимы напрямую при одинаковом числе воркеров.

Проверка честности: под нагрузкой оба стека реально выполняли работу (сотни тысяч
вставок в MongoDB/MySQL/PostgreSQL на прогон, все ответы `200` — упавшая фича с
2026-07-04 превращает ответ в 500, и таких не было) — а не отвечали вхолостую.

Строка `/all-native` — ручка SConcur-сервера: точная копия
RoadRunner-воркера (те же нативные драйверы, последовательно, без фич SConcur) внутри
SConcur-сервера. Она изолирует слой сервера/транспорта от драйверного стека.

| Ручка | Сервер                                          | Requests/sec | p50 / p90 / p99 | CPU avg / peak | MEM peak | к RoadRunner |
| --- |-------------------------------------------------| ---: | --- | --- | ---: | :---: |
| `/` (пусто) | SConcur                                         | ≈67 100 | 3.7 / 6.3 / 8.8 мс | ~1210% / ~1210% | ~256 MiB | +42% ✅ |
| `/` (пусто) | RoadRunner                                      | ≈47 100 | 5.3 / 5.9 / 6.7 мс | ~1060% / ~1075% | ~230 MiB | — |
| `/all` | SConcur (SConcur, асинхронно)                   | ≈2 680 | 87 / 165 / 267 мс | ~740% / ~765% | ~287 MiB | +483% ✅ |
| `/all` | RoadRunner (нативные драйверы, последовательно) | ≈460 | 561 / 589 / 603 мс | ~160% / ~175% | ~237 MiB | — |
| `/all-sconcur` | RoadRunner (SConcur, асинхронно)                | ≈583 | 442 / 469 / 522 мс | ~540% / ~600% | ~363 MiB | +32% ✅ |
| `/all-native` | SConcur (нативные драйверы, последовательно)    | ≈457 | 556 / 745 / 832 мс | ~140% / ~160% | ~265 MiB | — |

На пустой ручке SConcur быстрее ~1.4×: цена запроса через границу PHP↔Go низкая
(задачи уходят в Go со стека планировщика, канал результатов буферизован), а
RoadRunner платит IPC-хоп прокси → воркер на каждый запрос.

На `/all` с дисковыми бэкендами картина перевернулась радикально: **SConcur быстрее
RoadRunner в ~6×** (≈2 680 vs ≈460 rps). Причина — fsync: 3 записи на запрос у
последовательного воркера складываются в цепочку дисковых коммитов (p50 ≈ 0.56 с, CPU
простаивает на ~160%), и 12 воркеров упираются в неё все разом. Веер SConcur
накладывает эти же fsync'и друг на друга и на соседние запросы — кооперативный воркер
обслуживает десятки запросов, припаркованных на I/O, пока диск работает.

Строка `/all-native` подтверждает, что дело не в серверном слое: тот же нативный
последовательный код внутри SConcur-сервера даёт те же ≈457 rps, что RoadRunner
(≈460), с теми же латентностями — сервер/транспорт SConcur нативному стеку не
уступает, а шестикратный разрыв создаёт именно модель исполнения
(последовательные fsync'и против наложенных веером).

Строка `/all-sconcur` — обратный эксперимент: веер SConcur внутри
RoadRunner-воркера. Веер накладывает три fsync'а запроса друг на друга, и время
запроса стремится к максимуму из трёх, а не к сумме: +32% rps, p50 с 586 до
442 мс. Дальше этой схеме не уехать: воркер RoadRunner обслуживает один запрос за
раз, поэтому припарковать десятки запросов на I/O, как кооперативный
SConcur-сервер (≈2 680 rps), он не может — потолок остаётся «воркеры × время
запроса». Память воркеров растёт до ≈363 MiB на 12 воркеров (Go-рантайм
расширения в каждом воркере). Это сценарий «дополнить RoadRunner, а не заменить»:
существующее RR-приложение получает веерную конкурентность внутри запроса, не
меняя серверную модель. (Пара `/all-sconcur`/`/all` снята отдельной сессией
2026-07-24; native `/all` в ней дал ≈443 rps / p50 586 мс, и процент строки
посчитан против этого одноcессионного эталона.)

Граница применимости веера проходит по цене ожидания операции. Там, где её нет
(данные в памяти или горячем кэше на локальном loopback), последовательному нативному
коду накладывать нечего и он выигрывает — это видно по строкам дешёвых чтений в
таблицах БД. Как только у операции появляется задержка — fsync на диске, как здесь,
или сетевой RTT — последовательный код суммирует её, веер накладывает, и преимущество
уходит к SConcur.

## Выводы

- Одиночный вызов через SConcur (`sync`) всегда дороже родного драйвера — цена конвертации
  на границе PHP↔Go (MessagePack + BSON у MongoDB + cgo). Заметнее всего она на дешёвых
  чтениях из кэша, где сам запрос почти мгновенный.
- Выигрыш веера (`async`) прямо пропорционален цене операции — ожиданию I/O (fsync,
  сетевой RTT) или реальной серверной работе, — которую он накладывает, а не суммирует.
  На дисковом SQL выигрывает каждая запись: mysql `insert`/`update`/`delete` ~14–16× и
  `transaction` ~10×, pgsql `update` ~18×, `insert`/`delete` ~5–6× и `transaction` ~3×.
  На датасете в 100 тыс. выигрывают и тяжёлые чтения: pgsql `count` ~7.5×, mongo `count`
  ~7×, `updateMany` ~5.5×, `bulkWrite` ~6.5×, mysql `count` ~2×. За native остаются
  дешёвые точечные операции, где накладывать нечего: `selectOne`/`findOne`, `selectMany`
  (доминирует конвертация набора строк на границе) и однодокументные mongo-операции
  (`insertOne`, `updateOne`, `deleteOne` — без fsync на операцию при дефолтном write
  concern): там оверхед границы превышает саму операцию.
- Пул соединений решает: холодный пул стоил вееру 3–15× (открытие соединений внутри
  замера), поэтому в методологии есть прогрев, а `maxIdleConns` по умолчанию равен
  `maxOpenConns` — иначе Go держит 2 idle и пул схлопывается между веерами.
- С сетевой задержкой картина та же, что с fsync: на клиентах со 100 мс ожидания веер
  даёт ~44×.
- Граница PHP↔Go на веере дешёвая (задачи в Go уходят со стека планировщика, канал
  результатов буферизован): throughput socket/ws-серверов — ~120–154 тыс.
  round-trip/с, пустая HTTP-ручка — ~67 тыс. rps (~1.4× быстрее RoadRunner).
- Сравнение с RoadRunner на дисковых бэкендах: `/all` веером — ~6× быстрее
  последовательного нативного воркера (≈2 680 vs ≈460 rps); `/all-native` показывает,
  что серверный слой SConcur нативному стеку не уступает (≈457 ≈ RoadRunner). Внутри
  RoadRunner-воркера веер SConcur (`/all-sconcur`) даёт +32% rps и −25% p50 — потолок
  задаёт модель «один запрос на воркер». Проигрыш
  веера остаётся только там, где у операций нет реальной цены I/O (память/горячий кэш
  на loopback — строки дешёвых чтений в таблицах БД).
- Память практически одинакова во всех режимах: в таблицах БД пики режимов равны
  (`selectMany` +2 МБ на наборы строк) — фиберы корутин веера шириной 100 пик заметно
  не сдвигают.
