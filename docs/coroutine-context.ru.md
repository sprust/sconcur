# Контекст корутины

`SConcur\Context\Context` — key-value хранилище, привязанное к текущей корутине
(её фиберу). Оно изолировано между конкурентными корутинами и наследуется
дочерними. Нужно, чтобы переносить per-request состояние (запрос, пользователя,
локаль) через `suspend`/`resume`, не складывая его в общие синглтоны — соседняя
корутина их перетёрла бы.

Фича фреймворк-нейтральная: хранилище держит произвольные `mixed` по строковым
ключам и ничего о них не знает. Интеграция с конкретным фреймворком строится
поверх неё.

## API

```php
use SConcur\Context\Context;

$context = Context::current();      // контекст текущей корутины (вне фибера — корневой)

$context->set('user', $user);       // запись локально в текущую корутину
$context->find('user');             // значение или null (с учётом наследования)
$context->has('user');              // виден ли ключ (свой или унаследованный)
$context->forget('user');           // удалить только локальный ключ
```

`Context::current()` — статический вход. Возвращает `CoroutineContext`,
зафиксированный на id текущего фибера в момент вызова, поэтому ссылку можно
держать через собственные `suspend`/`resume` этой корутины.

Сигнатура `set`:

```php
public function set(string $key, mixed $value, bool $replace = false): void;
```

При `replace: false` (по умолчанию) уже существующий локальный ключ не
перезаписывается; `replace: true` перезаписывает.

## Семантика

Привязка к корутине идёт по `Fiber::getCurrent()` — тому же признаку, по которому
`State::getCurrentFlow()` резолвит flow. У каждой корутины (spawned-обработчик
запроса и каждая корутина `WaitGroup`) — своя локальная карта.

Чтение — read-through по цепочке родителей: сначала собственная карта, затем
родитель, и так до корня процесса. Запись (`set`/`forget`) — всегда локальная:
дочерняя может затенить ключ родителя, не трогая родителя и соседние корутины.

Родитель фиксируется в момент создания корутины: это корутина, в которой вызвали
`Scheduler::spawn` или `WaitGroup::add`. Корутина, созданная вне любого фибера
(обработчик запроса в цикле сервера, код бутстрапа), наследуется от корневого
контекста — он общий на процесс и живёт всё его время.

Вне фибера `Context::current()` возвращает корневой контекст, а не бросает
исключение, так что код инициализации (до первого запроса) работает штатно.

## Наследование во вложенных корутинах

Когда обработчик запроса порождает вложенные корутины (например, параллельные
запросы в БД через `WaitGroup`), они видят ключи родителя:

```php
Context::current()->set('request', $request);

$group = WaitGroup::create();

$group->add(static function () {
    // унаследует 'request' от породившей корутины
    $request = Context::current()->find('request');
    // ...
});

$group->waitAll();
```

## Жизненный цикл

Локальная карта создаётся лениво (при первом `set`) и освобождается вместе с
корутиной — там же, где библиотека снимает per-fiber учёт
(`State::unRegisterFiber`, через `Scheduler::forget`/`detach` и
`State::deleteFlow`). После завершения N корутин в памяти не остаётся N контекстов.
Корневой контекст не удаляется.

## Ограничения

- Среда кооперативная и однопоточная, блокировок нет.
- `find`/`has`/`set` — горячий путь: O(1) по собственной карте плюс подъём по
  цепочке родителей (глубина обычно 1–2: запрос → вложенная корутина).
- Объём значений — на стороне вызывающего; библиотека их не интерпретирует, но
  гарантирует освобождение по завершении корутины.
