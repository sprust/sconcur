# Сокет-сервер (TCP)

Долгоживущий TCP-сервер: сеть живёт в Go-расширении, каждое принятое соединение
стримится в PHP и обрабатывается в своей корутине. Внутри соединения обработчик
вызывается **на каждое входящее сообщение** (один length-prefix-фрейм) по порядку
и возвращает один ответ на сообщение.

Эталон по устройству — [HTTP-сервер](http-server.ru.md); сокет-сервер — это его
обобщение на много раундов «запрос/ответ» в одном соединении. Запускается под тем
же [мастером воркеров](worker-master.ru.md).

## Фрейминг сообщений

Поток байтов соединения нарезается на сообщения по схеме **length-prefix**: каждое
сообщение — это `uint32` big-endian длина payload, затем сам payload. Тот же формат
в обе стороны (и вход, и ответ). Это бинарно-безопасно (нулевые байты, любые
данные), не требует экранирования и даёт естественный лимит `maxMessageBytes`.

```
[len=5]hello[len=3]bye
```

Клиент обязан кадрировать так же. Пример на PHP:

```php
function sendFrame($connection, string $data): void
{
    fwrite($connection, pack('N', strlen($data)) . $data);
}
```

## Быстрый старт

```php
use SConcur\Features\SocketServer\Dto\Message;
use SConcur\Features\SocketServer\SocketServer;

$server = new SocketServer(address: '0.0.0.0:9100');

$server->serve(static function (Message $message): string {
    // echo: вернуть тот же payload
    return $message->data;
});
```

Сервер слушает вечно — пока не придёт сигнал остановки (SIGTERM/SIGINT) или поток
не будет остановлен.

## Обработчик

Сигнатура: `Closure(Message): (string|MessageResponse|null)`.

`Message` (`src/Features/SocketServer/Dto/Message.php`):

| Поле | Описание |
| --- | --- |
| `connectionId` | сквозной id соединения (можно держать per-connection состояние в userland) |
| `data` | payload сообщения (бинарно-безопасно) |
| `remoteAddr` | адрес клиента `ip:port` |
| `localAddr` | локальный адрес |
| `messageIndex` | порядковый номер сообщения в соединении (с 0) |

Возврат обработчика:

- **`string`** — отправить ответный фрейм (включая пустую строку — это валидный
  фрейм нулевой длины), соединение остаётся открытым;
- **`null`** — ответа нет, соединение остаётся открытым (fire-and-forget);
- **`MessageResponse`** — ответить и/или закрыть соединение:
  ```php
  use SConcur\Features\SocketServer\Dto\MessageResponse;

  return new MessageResponse(data: 'bye', close: true); // ответить и закрыть
  return new MessageResponse(close: true);              // закрыть без ответа
  ```

Обработчик исполняется в корутине соединения, поэтому внутри можно делать
асинхронные вызовы (Sleeper, Mongodb, SQL, HTTP-клиент) — корутина кооперативно
приостанавливается, другие соединения продолжают обслуживаться.

### Состояние соединения: `onConnect` / `onClose`

Опциональные хуки конструктора для per-connection состояния:

```php
$sessions = [];

$server = new SocketServer(
    onConnect: function (string $connectionId, string $remoteAddr) use (&$sessions): void {
        $sessions[$connectionId] = ['since' => time()];
    },
    onClose: function (string $connectionId) use (&$sessions): void {
        unset($sessions[$connectionId]); // гарантированно вызывается при любом завершении
    },
);
```

### Обработка ошибок: `onError`

Если обработчик бросил исключение или вернул не тот тип, по умолчанию ответ **не
отправляется** и соединение **не закрывается** (на сокете нет понятия «500»). Хук
`onError: Closure(Throwable, Message): ?string` позволяет это пронаблюдать и
опционально вернуть строку-ответ:

```php
$server = new SocketServer(
    onError: function (Throwable $exception, Message $message): ?string {
        error_log($exception->getMessage());

        return "error\n"; // или null — не отвечать
    },
);
```

## Параметры

Конструктор `SocketServer` (значения по умолчанию зеркалят Go):

| Параметр | По умолчанию | Назначение |
| --- | --- | --- |
| `address` | `0.0.0.0:9100` | адрес слушателя `host:port` |
| `readTimeoutMs` | `0` (выкл) | idle-таймаут между сообщениями: нет нового фрейма за это время → соединение закрывается |
| `writeTimeoutMs` | `30000` | максимум на запись одного ответного фрейма |
| `handlerTimeoutMs` | `60000` (0 = выкл) | максимум на обработку одного сообщения; таймер на Go-стороне, независим от PHP |
| `maxMessageBytes` | `1048576` (1 MiB) | лимит длины одного входящего фрейма; превышение закрывает соединение |
| `maxConcurrency` | `0` (без лимита) | максимум одновременно обслуживаемых соединений; лишние ждут свободный слот |
| `maxConnections` | `0` (без лимита) | остановить сервер после N обслуженных соединений (мера против утечек) |
| `shutdownTimeoutMs` | `5000` | таймаут дренажа in-flight соединений при остановке |
| `reusePort` | `false` | `SO_REUSEPORT` — пул процессов на один порт (Linux) |
| `onError` | `null` | хук ошибки обработчика |
| `onConnect` | `null` | хук открытия соединения |
| `onClose` | `null` | хук закрытия соединения |
| `masterPid` | `null` | orphan-чек под мастером (см. ниже) |

## Конкурентность и порядок

Конкурентность — **между соединениями**: каждое соединение обрабатывается в своей
корутине, поэтому десятки соединений работают параллельно. **Внутри одного
соединения** сообщения обрабатываются строго **по порядку** (read → handler →
ответ → следующее), что и требует протокол с length-prefix без корреляционных id.

`maxConcurrency` ограничивает число одновременно обслуживаемых соединений (слот
держится всё время жизни соединения); лишние соединения принимаются на сокете, но
не обрабатываются, пока не освободится слот.

## Таймаут обработчика

`handlerTimeoutMs` ограничивает время обработки **одного сообщения**. Таймер
работает на Go-стороне, **независимо от PHP**, поэтому он закрывает соединение даже
если PHP-поток заблокирован нативно (например, блокирующий `usleep`/CPU-цикл).
Замечание: CPU-bound обработчик всё равно блокирует единственный поток PHP — таймаут
страхует обработчики, ждущие асинхронной работы.

## Graceful shutdown и SO_REUSEPORT

По сигналу (SIGTERM/SIGINT), по достижении `maxConnections` или при сиротстве
(`masterPid`) сервер уходит в graceful-дренаж: перестаёт принимать новые соединения
(закрывает слушатель) и **полузакрывает** in-flight соединения на чтение — текущий
ответ ещё дописывается, а простаивающие соединения завершаются. На пуле
`SO_REUSEPORT` ядро тут же раздаёт новые соединения соседним процессам, пока этот
дренажит, после чего процесс завершается сам.

`reusePort: true` позволяет нескольким процессам слушать один порт (один процесс на
ядро) — основа масштабирования под мастером воркеров.

## Запуск под мастером воркеров

Сервер — «server-agnostic»-воркер для `bin/sconcur-server`. Пример конфига —
`config/sconcur.socket-server.config.json`; воркер-скрипт строит сервер из argv:

```php
use SConcur\Features\SocketServer\Dto\Message;
use SConcur\Features\SocketServer\SocketServer;

$server = SocketServer::fromArgs($_SERVER['argv']);

$server->serve(static fn (Message $message): string => $message->data);
```

Параметры из блока `server` JSON-конфига мастер разворачивает в `--ключ=значение`
argv (`fromArgs` их разбирает), а свой pid прокидывает флагом `--masterPid`
(orphan-чек: воркер сам уходит в дренаж, если мастер умер). `reusePort: true`
включает пул процессов на ядра. Подробности — в [мастере воркеров](worker-master.ru.md).

## Ограничения

- **Только TCP.** Unix-сокеты в этой версии не поддерживаются (`SO_REUSEPORT`
  неприменим к `AF_UNIX`; мультиворкеры для unix требуют наследования fd — отдельная
  задача).
- **Порядок внутри соединения последовательный.** Конвейеризация (несколько
  ответов «в полёте» на одном соединении) не поддерживается — ответы идут строго по
  одному на сообщение, по порядку.
- Общие ограничения библиотеки (только CLI, только Linux, только NTS, нельзя
  `pcntl_fork` после загрузки расширения) — см. [README](../README.md).
