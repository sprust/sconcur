# Производительность моста PHP ↔ Go (протокол обмена и налог на вызов)

> **Статус: тема закрыта по результатам профиля (2026-07-02).** Идея **A** (framing
> результата — msgpack только для payload) **реализована**. Профиль показал, что у
> оставшихся glue-идей мало запаса: userland pack/unpack + cgo ≈ 1% налога («зерно» и
> **C** — мимо), а обобщённый налог обвязки — единицы микросекунд на вызов (потолок **D**
> < ~8 мкс, **B** ничтожна). Ощутимо это только на операциях с околонулевой задержкой, где
> нативному коду и так не проиграть; для реальных фич доминирует I/O + Go-драйвер. Вывод:
> дальше оптимизировать обвязку **не имеет смысла** — документ остаётся картой/обоснованием
> на будущее, не планом работ. Детали — в разделах «Профиль» ниже.

Единственное реально **не измеренное** узкое место — **PHP-сторона веерного пути** (`/all`):
спавн Fiber, цикл `Scheduler`, вложенный `WaitGroup`. Весь профиль до сих пор — Go-сторона и
userland pack/unpack; разрыв SConcur vs RoadRunner на `/all` (≈2 580 vs ≈5 700 rps) им не
объясняется (Go-glue дёшев). Если что и профилировать дальше — это PHP-сторона веера, и уже
по её результатам решать про F. Остальное закрыто.

Сквозной план по ядру: как удешевить один вызов через границу PHP↔Go. Касается
**всех** фич (sleeper / mongodb / http / sql), а не одной. Меняет протокол и
экспортируемые функции → требует **бампа версии расширения**, прогона регресса по
всем фичам и **согласования с мейнтейнером** (правки ядра).

Контекст: на localhost БД/операция отвечает за доли мс, поэтому в бенчмарках виден
оверхед самой обвязки. На синхронном пути он ≈0.28 мс/вызов (см. план
[`mysql.md`](mysql.md), раздел «Производительность»). Async-путь амортизирует
оверхед конкурентностью, поэтому приоритет — синхронный путь и общий объём
копий/сериализаций.

## Текущий протокол (как есть)

- `sconcur.so` — рукописное PHP-расширение (`ext/sconcur.c`), регистрирует функции
  `SConcur\Extension\{push,next,wait,waitAny,waitAnyTimeout,stopFlow,…}`; каждая
  вызывает cgo-экспорт из `ext/main.go`.
- **PHP → Go (`push`)**: аргументы передаются по отдельности — `flowKey` (string),
  `method` (int), `taskKey` (string), `payload` (msgpack-байты как PHP-строка). Go
  копирует их внутрь (`C.GoStringN`, `C.GoBytes`). Конверт здесь **не** msgpack'ится
  — это эффективно.
- **Go → PHP (`wait`/`waitAny`)**: Go берёт `*dto.Result`, делает
  `msgpack.Marshal(res)` (рефлексия по структуре), `C.CBytes` (malloc+copy),
  возвращает `buffer_result_t{data,len,err}`. C-glue копирует в `zend_string`
  (`RETVAL_STRINGL`) и `free`-ит. PHP затем `msgpack_unpack`-ает конверт и собирает
  `TaskResultDto`.

### Путь данных одного sync-вызова (число операций)

1. PHP: `getData()` → `msgpack_pack` (сериализация payload). 
2. cgo: копия payload PHP→Go (`C.GoBytes`).
3. Go: обработка фичи → `*dto.Result` (поле `Payload` — уже готовые байты фичи).
4. Go: `msgpack.Marshal(Result)` (**вторая** сериализация — конверт + вложенный payload).
5. cgo: `C.CBytes` (malloc+copy Go→C).
6. C-glue: копия C→`zend_string`.
7. PHP: `msgpack_unpack` конверта (**десериализация**).
8. PHP: фича ещё раз `unpack`-ает поле `pl` (**двойная** десериализация payload).
9. Плюс 3 cgo-перехода на sync-вызов: `push` + `wait` + `stopFlow`.

## Узкие места

1. **Двойная сериализация на пути результата.** `dto.Result` маршалится msgpack'ом
   целиком, хотя его поле `Payload` — уже закодированные байты фичи. На PHP-стороне
   конверт распаковывается, затем payload распаковывается ещё раз. Это лишняя
   рефлексия в Go + лишний `unpack` в PHP + вложенная копия. **Асимметрия**: `push`
   передаёт конверт отдельными аргументами (дёшево), а `wait` заворачивает в msgpack
   (дорого).
2. **msgpack через рефлексию в Go** (`vmihailenco/msgpack/v5`) на каждый результат —
   медленнее кодогенерации/ручного кодека.
3. **3 cgo-перехода** на sync-вызов (`push`+`wait`+`stopFlow`) + накладные cgo.
4. **Аллокации на вызов**: `C.CString/CBytes` (malloc), `zend_string`, структуры
   сообщения/результата, одноразовый `Flow`+канал+контекст, spawn горутины.

## Идеи изменений (по убыванию «выгода/риск»)

### A. Убрать двойное кодирование результата (framing) — главный чистый выигрыш

Возвращать результат как **дешёвый бинарный заголовок + сырые байты payload**, без
msgpack на конверте и без повторной распаковки payload:

```
[flags:1][method:2][execMs:4][tkLen:2][tk bytes][fkLen:2][fk bytes][payload bytes…]
```
- `flags`: бит `isError`, бит `hasNext`.
- Go (`wait`/`waitAny`) пишет заголовок в буфер и **дописывает payload как есть** —
  без `msgpack.Marshal(Result)`.
- PHP (`Extension::parseWaitResponse`) читает заголовок через `unpack()`/`substr`,
  payload берёт срезом и отдаёт фиче, которая распаковывает его **один раз**.

Эффект: минус одна полная msgpack-сериализация (Go, рефлексия), минус одна
распаковка конверта (PHP), минус двойная распаковка payload, минус вложенная копия.
Симметрия с `push`. Затрагивает `main.go` (wait/waitAny/waitAnyTimeout), `sconcur.c`
(не обязательно — буфер уже бинарный), `Extension.php`. `TaskResultDto` по форме не
меняется. Выигрывают **все** фичи и async-путь. Риск средний (узкая точка), нужен
аккуратный регресс.

### B. Кодеген/ручной кодек вместо рефлексии (если оставляем msgpack где-то)

Для горячих структур (`Message` на разборе `push`, и результат, если A не делать)
использовать `msgpack.Marshaler`/`Unmarshaler` вручную или кодоген (`tinylib/msgp`),
чтобы убрать рефлексию. Меньший выигрыш, чем A; имеет смысл, если framing-подход A
не выбран или для пути `push` (разбор `Message`).

### C. Сократить число cgo-переходов (часть «Фазы 3»)

- **Слить `push`+`wait`** для sync-пути в один экспорт `pushWait(...)` → 3 cgo → 2.
- **Убрать отдельный `stopFlow`** на sync-пути: авто-реап одноразового flow в момент
  доставки финального результата (правка `FeatureExecutor::handleSync` + lifecycle
  `Flow`). 2 → 1. Итог: один cgo на sync-вызов.

### D. Не плодить flow/горутину на sync-вызов

- Переиспользовать пул flow'ов / один персистентный sync-flow (без аллокации
  канала+контекста+teardown на вызов).
- Радикально: для чисто синхронного пути выполнять обработчик **инлайн** на cgo-нити
  (без горутины и канала), возвращая результат прямо из `pushWait`. Самый большой
  выигрыш по «налогу», но самая инвазивная правка (расходится с моделью «всё через
  планировщик»); только для не-Fiber пути.

### E. (радикально) Разделяемая память / ring buffer

SPSC/MPSC-кольцо в shared memory: PHP пишет запросы и читает результаты, cgo — только
для пробуждения/блокировки. Убирает копии и часть cgo. Очень сложно (синхронизация,
владение памятью, GC Go vs zend_string), и cgo — не доминирующая статья. Скорее
исследование на будущее, чем ближайший шаг.

### Отклонённое

- **Zero-copy payload PHP→Go** (передать указатель zend_string без копии): опасно
  (жизненный цикл/перемещение строки, GC). Копия дешевле риска; payload мал.
- **FFI вместо C-расширения**: текущее рукописное расширение — уже самый быстрый путь
  вызова; FFI медленнее. Оставляем C-расширение.
- **FlatBuffers/Cap'n Proto** вместо msgpack глобально: zero-copy чтение, но кодоген +
  ридер на PHP-стороне и переписывание всех payload'ов. Для **конверта** дешевле
  ручной framing (A); для **payload'ов** msgpack приемлем. Не оправдано целиком.
- **zval-мост: передавать `mixed` в расширение, парсить zval в Go, результат собирать
  прямо в zval** (убрать msgpack совсем). Технически принять массив/`mixed` можно —
  `push` уже рукописный (`ext/sconcur.c:115`, сейчас `"slss"`). Но в чистом виде
  проигрывает по трём причинам:
  1. **cgo штрафует за мелкие частые вызовы, а не за копию.** Чтение zval «по полю» из
     Go через Zend API (обход `HashTable`, `Z_TYPE_P`…) — десятки cgo-переходов на
     вложенный payload, это анти-паттерн cgo. Текущая схема (один переход + один
     непрерывный msgpack-буфер + нативный декод) специально cgo-дружелюбна.
  2. **Асинхронность требует синхронной глубокой копии в любом случае.** `push`
     возвращается сразу, задача исполняется в горутине позже — к этому моменту PHP-фрейм
     освободит zval. Значит payload обязан быть материализован в Go-память синхронно во
     время `push`. Это ровно то, что делает `msgpack_pack` (обход zval в C → буфер);
     копию не убрать, можно лишь сменить формат.
  3. **zval Go не создаёт** (Zend MM + рефкаунты) — результат всё равно собирает
     C-сторона: либо из буфера (как сейчас), либо Go гоняет C-билдеры «по узлу» (снова
     много cgo). Первое дешевле.
  Вариант «C-сторона обходит zval и пишет буфер» — это переизобретение msgpack в своём
  C-коде; `msgpack_pack` уже и есть зрелый «обход zval в C → буфер». Вариант «мапить
  layout zval/HashTable/zend_string в unsafe-Go» — хрупко (версия PHP, NTS/ZTS, debug),
  опасно (рефкаунты, packed vs hashed, interned strings, GC Go). Отклоняем.
  - **Зерно, которое стоит взять** (не ломает cgo-дружелюбность): перенести сам
    pack/unpack из PHP-userland **в C** через C-API ext-msgpack — `push` принимает
    массив (`"a"`), сериализует внутри расширения; на ответе C-glue строит zval из
    Go-буфера, минуя userland `msgpack_pack`/`msgpack_unpack`. Транспорт остаётся одним
    буфером. Выигрыш — минус два userland-вызова и промежуточные строки на запрос;
    небольшой, риск низкий-средний. Меньше, чем B/F/H, — брать только после профиля.

## Дополнительно: веерные ручки (из сравнения с RoadRunner, 2026-07-02)

Замер `/all` (одна ручка гоняет 4 backend-операции: sleeper + Mongo insert/findOne +
MySQL/PgSQL `SELECT 1`) против RoadRunner на нативных драйверах показал: на in-memory /
локальном loopback RoadRunner быстрее ~2.2× при вдвое меньшем CPU (см.
[`docs/benchmarks.ru.md`](../../docs/benchmarks.ru.md), раздел про серверы). Причина —
не БД (они мгновенны), а фан-аут: на каждый запрос SConcur делает 4 пересечения границы
+ вложенный `WaitGroup` + спавн фиберов. Идеи A–E удешевляют **один** вызов; ниже —
углы, снижающие оверхед именно при **множестве операций на запрос**. Всё это имеет смысл
только когда латентность операции мала (localhost/in-memory); на реальной сетевой/дисковой
задержке оверхед обвязки и так пренебрежим — сначала стоит убедиться, что цель того стоит.

### Сначала — атрибуция налога (перед любой из идей ниже)

Прежде чем оптимизировать, разложить оверхед вызова на три статьи: сам cgo-переход,
msgpack pack/unpack, аллокации/сборка zval на PHP-стороне. Лечатся по-разному; профиль
(Go pprof + PHP `perf`) на голом sync-вызове и на `/all` покажет доминирующую статью.
Скорее всего это не сам msgpack, а число переходов (для `/all`) и zval-churn.

### F. Батч нескольких операций на одно пересечение (главный рычаг для веера)

Слать N независимых операций одним `push` и забирать N результатов одним кадром
(pipeline-конверт), вместо N отдельных `push`/`wait`/фиберов. Амортизирует cgo + копию +
pack на пачку — прямое лекарство для `/all`-подобных ручек, где сейчас 4-кратный налог.
Минус: теряется независимое планирование/стриминг по каждой операции, поэтому реализуется
как **отдельный** batch/pipeline-путь поверх текущего per-op, не ломая его. Ортогонально
C (C сливает push+wait для одной операции; F складывает много операций в один переход).

### G. Framing запроса и позиционные кортежи для дешёвых payload'ов

Расширить идею A на путь `push`: для тривиальных payload'ов (sleeper — один int; SQL с
парой биндингов) не msgpack-map с хешированием ключей, а фиксированный бинарный фрейм или
позиционный msgpack-массив `[cm, args…]`. Меньше байт и декода без хеширования строковых
ключей. Затрагивает форму `getData()` фич и разбор `Message` на Go-стороне.

### H. Ленивый/типизированный декод результата на PHP

Возвращать результат, декодирующий поля по обращению, вместо жадной сборки больших
вложенных массивов на каждый ответ (снижает zval-churn — вероятную доминанту на дешёвых
операциях). Для выборок — стриминг строк с ленивым декодом (частично уже есть через
`next()`).

### I. Считать на стороне Go, чтобы данные не пересекали границу

Обобщить паттерн download (тело пишется в файл на Go, в PHP не заходит): если PHP нужен
только итог (кол-во, статус, хеш, инкремент, bulk-результат) — считать в Go и вернуть
маленький результат вместо перекачки полного набора. Убирает и копию, и распаковку.

### Порядок для веерного сценария

1. **Атрибуция** (профиль cgo/pack/zval на `/all` и голом вызове).
2. **F — батч операций** (единственная мера с крупным эффектом для веера).
3. **G / H** — дешёвые добавки по результатам профиля.
4. **I** — точечно, где PHP не нужен полный payload.

## Профиль: атрибуция налога (2026-07-02)

Замер на localhost (PHP 8.4.15, extension debug off), мкс на вызов, лучшее из прогонов.
«Голый налог» — sync sleeper с `usleep(1)` (без БД); userland-примитивы — в тесном цикле.

| Что | мкс/вызов |
| --- | ---: |
| cgo-переход (один, `tasksCount()`) | 0.082 |
| `msgpack_pack` (sleeper, 7 Б) | 0.106 |
| `msgpack_pack` (SQL SELECT 1, 110 Б) | 0.215 |
| `msgpack_unpack` (результат, 7 Б) | 0.171 |
| **Голый sync-налог** (sleeper `usleep(1)`, без БД) | **40.5** |
| SELECT 1 sync (с БД по loopback) | 119 |

Вывод — приоритеты меняются:

- **Идея «зерно» (pack/unpack в C) и идея C (слияние cgo) — мимо.** userland pack+unpack +
  3 cgo-перехода = `0.106 + 0.171 + 3×0.082 ≈ 0.5 мкс` из 40.5 (~1.2%). Переносить это в C
  — оптимизировать ~1% налога, ещё и добавив жёсткую зависимость от ext-msgpack (символы
  `php_msgpack_serialize`/`php_msgpack_unserialize` есть в `msgpack.so`, но нужен
  `RTLD_GLOBAL` и порядок загрузки). Не оправдано.
- **Реальный налог (~40 мкс) — per-call машинерия на Go-стороне**: спавн горутины,
  создание/тир-даун канала и одноразового `Flow`, планирование, unmarshal `Message`. Это
  ровно **идея D** (переиспользование flow / инлайн-исполнение sync-пути без горутины и
  канала). Сюда же частично **B** (unmarshal `Message` — рефлексия), долю которого стоит
  измерить Go-профилем отдельно.

### Go-профиль (2026-07-02): «40 мкс» оказались таймером, а не glue

Бенчмарк пути `handler.Push`+`Wait`+`StopFlow` (без cgo/PHP, `CGO_ENABLED=0`) с изоляцией
компонент. Правит вывод выше: sleeper — **плохой прокси** голого налога, его тащит
собственный таймер.

| Изоляция | мкс/op | аллокации |
| --- | ---: | --- |
| `<-time.After(1µs)` (собственное ожидание sleeper) | ~33 | 3 |
| спавн горутины + rendezvous по небуф. каналу | ~0.5 | 2 / 128 Б |
| unmarshal payload (`{us:1}`) | ~0.19 | 2 / 56 Б |
| full sleeper round-trip, fresh flow | ~48–68 (шумно) | 21 / 1586 Б |
| full sleeper round-trip, reused flow | ~42–48 | 13 / 785 Б |

- «~40 мкс» из PHP-замера `usleep(1)` — на **~80% таймер самого sleeper** (~33 мкс на
  пробуждение 1µs-таймера Go), а не обвязка. Sleeper нельзя брать как прокси glue-налога.
- Обобщённый налог glue = rendezvous (~0.5) + unmarshal (~0.19) + одноразовый `Flow`
  (fresh−reused: ~8 аллокаций / ~800 Б) ≈ **единицы микросекунд CPU на вызов**. ns/op-дельта
  fresh vs reused тонет в шуме планировщика; надёжный сигнал — аллокации.
- Потолок **идеи D** (убрать горутину/канал/одноразовый flow) ≈ **< ~8 мкс/вызов**, а не 40.
  **B** (msgpack-декод) по латентности не оправдана — 0.19 мкс/вызов (те «26% CPU» в первом
  заходе — от отдельного unmarshal-бенча на 10.7 млн итераций, не от round-trip).

Итог: у glue-оптимизаций мало запаса (< ~10 мкс/вызов) и только на операциях с околонулевой
задержкой — там, где нативному коду и так не проиграть. Для реальных фич (SQL/mongo/http)
доминирует I/O + Go-драйвер, вне обвязки. D остаётся единственной осмысленной, но скромной
мерой — делать её только если единицы микросекунд стоят инвазивной правки ядра и бампа
версии.

## Процесс и риски

- Правки ядра и протокола → **бамп версии расширения** и согласование с мейнтейнером
  (по правилам — мажор только с одобрения).
- Полный регресс по **всем** фичам (sleeper/mongodb/http-server/http-client/sql) и по
  async-пути (главная ценность библиотеки) — нельзя ломать конкурентность.
- Гейт: `make ext-build && make ext-test && make php-stan && make cs-fixer-check &&
  make test` + бенчмарки до/после (`bench-all`), измерять **лучшее из N** прогонов
  из-за разброса на localhost.

## Как измерять

- Контролируемый замер sync exec/select на N=1000, лучшее из 3 (как в Фазе 1).
- Отдельно мерить «голый налог»: фиктивная операция без I/O (или sleeper c
  минимальной длительностью) — чтобы видеть оверхед обвязки в чистом виде до/после.
- Async-путь: `bench-all` (insert/select/… native/sync/async) — следить, что async
  не регрессирует.
