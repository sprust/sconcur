# Небатчинговая оптимизация координации веера: цена park/wake на `waitAny`

> **Статус: предложение (2026-07-24), в работе с фазы 1.** Небатчинговая
> альтернатива плану [`php-go-boundary-batching.md`](php-go-boundary-batching.md):
> та же адресуемая статья (⅔ потолка throughput, теряемые на round-trip'ах
> `waitAny` под конкуренцией), но лечится **не** уменьшением числа переходов
> (батч), а **устранением стоимости парковки/пробуждения** PHP-потока на каждом
> переходе. Опирается на профили [`php-go-bridge-performance.md`](php-go-bridge-performance.md)
> и [`async-fan-out-optimization.ru.md`](async-fan-out-optimization.ru.md).
> Батчинг по условию задачи не рассматриваем.

## Мотивация: неатрибутированный разрыв

Профили закрыли «glue-идеи» (cgo-переход `0.082 мкс`, `msgpack pack/unpack
≈ 0.1–0.2 мкс`, framing результата — идея A реализована). Но в них же осталась
несостыковка, которая и есть цель этого плана:

- **Координация веера ≈ 130 мкс CPU на приостанавливающуюся корутину**
  (`suspend → waitAny → resume`), при сумме примитивов ~2 мкс — разрыв **~128 мкс
  на корутину не атрибутирован**.
- **Пустой веер (корутины не паркуются) — ~11 мкс/корутина**, реальный веер —
  ~130. Разница ≈ прямая цена **парковки PHP-потока и его пробуждения через
  границу**.
- **Потолок throughput `/all`:** CPU/req даёт расчётные ~7900 rps, измерено
  **~2580 (~⅓)**; авторы профиля списывают остаток на «round-trip'ы
  `waitAny`/планировщика под конкуренцией». Это единственная крупная статья, не
  разложенная до конца.

**Ключевая гипотеза.** Прошлые микробенчи промахнулись измерить именно эту статью:
они мерили либо **неблокирующий** cgo (`tasksCount`, 0.082 мкс), либо изолированное
рандеву при `CGO_ENABLED=0` (~0.5 мкс) — то есть **без реального парка cgo-нити и её
пробуждения Go-планировщиком с другого OS-потока**. В проде же `waitAny` —
блокирующий cgo `select` на канале (`ext/internal/handler/handler.go:62`): при
пустом буфере PHP-поток паркуется на futex, и завершившаяся горутина БД должна
разбудить cgo-locked поток через границу. Эти park/wake round-trip'ы и съедают ⅔
потолка.

**Корень.** PHP-сторона строго однопоточна и сериализует все результаты через одну
блокирующую точку `waitAny`. Go параллелит на N ядер, а результаты втягиваются по
одному с парковкой. Батчинг режет *число* переходов; этот план режет *стоимость
парковки на каждом переходе* — ортогональная и нетронутая ось.

**Оговорка о свежести цифр.** После профиля 2026-07-02 канал результатов стал
буферизованным (`resultsBufferSize = 1024`, `ext/internal/handler/handler.go:23`),
чтобы убрать rendezvous «две futex-побудки на результат». Поэтому цифры 130/⅓ могли
измениться, и план начинается с обязательной атрибуции (фаза 0), гейтящей всё
остальное — как замеры раньше закрыли идеи B/C.

## Цель

1. Атрибутировать разрыв ~128 мкс/корутина числом на сегодняшнем (буферизованном)
   коде.
2. Снять цену park/wake на горячем пути `waitAny` — без изменения протокола там, где
   это возможно (фаза 1), и радикально через shared memory (фаза 2).
3. Срезать per-call аллокации → давление на Go GC → CPU под серверной нагрузкой
   (фаза 3).
4. Точечно убрать перекачку данных через границу, где PHP нужен только итог
   (фаза 4).

## Честная рамка

- На in-memory single-row операциях паритет с native правками координации не
  гарантирован: серийная PHP-цена операции (pack → переход → resume → unpack →
  сборка zval) сопоставима со стоимостью всей нативной операции. Реальная цель —
  поднять потолок throughput под конкуренцией (закрыть ⅔-разрыв) и снизить CPU, а не
  победить native на голом `SELECT 1`.
- Выигрыш растёт с шириной веера и числом одновременных корутин: чем больше park/wake
  событий в секунду, тем больше отдача.
- Каждая фаза — отдельный измеримый шаг с замером «до/после» (лучшее/медиана из N
  прогонов) и регрессом по всем фичам.

## Фаза 0 — атрибуция park/wake (гейт для всего остального)

Замерить сегодняшнюю цену одного результата через `waitAny`, разложив на:

1. **Go-сторона, пустой vs полный буфер.** Go-бенч (`CGO_ENABLED=0`,
   `ext/internal/handler`) `Handler.WaitAny`: (а) буфер заполнен — быстрый
   неблокирующий receive; (б) буфер пуст — реальная парковка на канале + пробуждение
   горутиной-продьюсером с другого P. Мерить ns/op и futex-побудки
   (`perf stat -e ...` или `GODEBUG=schedtrace`).
2. **Реальный cgo park/wake из PHP.** Микробенч: веер `Sleeper::usleep(1)` такой
   ширины, что PHP-поток стабильно опустошает буфер и паркуется; wall+CPU через
   `getrusage` на один round-trip. Сравнить с тем же веером, но заранее завершёнными
   задачами (буфер всегда полон — парка нет). Дельта = чистая цена park/wake.
3. **Разбивка PHP-стороны на результат:** `parseWaitResponse` + `TaskResultDto`
   (`src/Connection/Extension.php:201`) отдельно от `resumeByResult`
   (`src/Scheduler/Scheduler.php:460`).

**Критерий гейта.** Фазы 1–2 делаются, только если цена park/wake (п.1–2) —
заметная доля round-trip'а под конкуренцией (ориентир ≥ ~10%). Иначе закрыть их как
закрыты B/C и дополнить карту в `php-go-bridge-performance.md`.

**Инструмент.** Готовые харнессы: `make bench-db-lifecycle`, `bench-all`,
`bench-http-load-stats` (`/all`), плюс профильный скейлинг `Sleeper::usleep(1)` по
N из `async-fan-out-optimization.ru.md`.

#### Результаты (2026-07-24, контейнер, машина занята сторонними контейнерами → шум)

Временный харнесс: два режима веера `Sleeper::usleep(1)`, per-op wall+CPU
(`getrusage`), лучшее из 3.

| Сценарий | wall мкс/оп | cpu мкс/оп |
| --- | ---: | ---: |
| Насыщенный веер (буфер полон, парка почти нет), N=1000–8000 | 8–12 | 8–16 |
| Последовательная цепочка (буфер всегда пуст, парк на каждом оп), N=2000–8000 | 20–50 | 28–46 |
| Far-трикл (2 корутины × `usleep(2000)`, результат ~2 мс) — baseline | 1127 | **142** |

Выводы атрибуции:

1. **Буфер канала (`resultsBufferSize=1024`) уже забрал основную цену park/wake на
   насыщенной нагрузке** — там поток не паркуется (wall ≈ cpu ≈ 8–16 мкс/оп),
   премисса «130 мкс/корутина» из профиля 2026-07-02 к сегодняшнему коду не
   применима: буфер её закрыл (см. `async-fan-out-optimization.ru.md`, п.2).
2. **Park/wake реально стоит** ~30 мкс wall + ~20–30 мкс CPU на оп, **но только когда
   поток действительно паркуется** — трикл/низкая конкуренция, где буфер пустеет
   быстрее, чем горутины его наполняют.
3. Насыщенный веер — это и есть сценарий throughput, ради которого оптимизируем; он
   уже быстр. Трикл — низкоконкурентный, там +30 мкс латентности незаметны на фоне
   реального I/O, которого ждёт приложение.

Гейт: адресуемая статья (park/wake) заметна только в трикле — фаза 1 проверяется,
но её потолок ограничен именно трикл-сценариями.

## Фаза 1 — spin-before-park в `waitAny` (без протокола, без бампа)

Самый дешёвый рычаг: под веерной нагрузкой результат приходит в течение
микросекунд, но `waitAny` при пустом буфере сразу паркует поток. Перед блокирующим
`select` крутить короткий **адаптивный спин** неблокирующих `select { case
<-results: default: }` (несколько итераций с `runtime.Gosched()`), и только потом
падать в блокирующий `select`. Если результат успевает прийти в окне спина — парка и
futex-пробуждения нет вовсе.

- Правка — только `ext/internal/handler/handler.go` (`WaitAny`); протокол,
  экспортируемые функции и PHP-сторона не меняются → **бампа версии нет**.
- Спин — только в `WaitAny` (горячий путь планировщика), **не** в `WaitAnyTimeout`
  (это idle-poll серверного цикла на 250 мс — спин там жёг бы CPU на простое).
- Число итераций спина — тюнируемая константа `waitSpinIterations`; подобрать по
  фазе 0 (баланс «CPU спина» vs «цена парковки»). Начать с малого (напр. 30–100
  итераций с Gosched) и мерить.
- Устаревшие результаты (`deliver()` вернул false из-за остановленного flow) в
  спине дренируются так же, как в блокирующем цикле.

**Риск:** спин — это CPU в обмен на латентность. Если результаты приходят *реже*
окна спина (узкий веер, редкие события), спин жжёт CPU впустую. Поэтому: (а) спин
короткий; (б) замер CPU обязателен — метрика успеха включает «CPU не вырос на
редких/idle сценариях». Возможен адаптивный спин (backoff по недавней истории
попаданий), если фиксированный даёт регресс CPU на узких веерах.

**Критерий успеха:** rps `/all` и `socket-throughput` растут, p50 падает, CPU/req не
растёт (в идеале падает за счёт снятых futex-побудок); `bench-all` без регресса.

#### Результаты (2026-07-24) — реализовано, эффект маргинальный

Реализация: `ext/internal/handler/handler.go`, `WaitAny` — спин перед блокирующим
`select`. Ключевой урок по механике спина:

- **`runtime.Gosched()` на cgo-locked нити стоит ~4 мкс** за итерацию. Спин из 256
  Gosched на far-трикле раздул CPU **142 → 1190 мкс/оп (×8)** — недопустимо: поток
  спинит ~1 мс на каждый парк, когда результат далеко.
- **Busy-poll без Gosched** (2048 итераций) far-трикл не ломает (CPU ~180), но и
  трикл-латентность не чинит: результат приходит дальше наносекундного окна, спин
  промахивается и всё равно паркуется.
- **Компромисс — `waitSpinIterations = 8` Gosched.** Достаточно уступок
  планировщику, чтобы горутина-продьюсер успела доставить близкий результат, с
  ограниченной ценой CPU:

| Сценарий | baseline | Gosched-8 |
| --- | ---: | ---: |
| Насыщенный веер, wall/cpu мкс/оп | 8–16 | 8–16 (нейтрально) |
| Последовательная цепочка, wall мкс/оп | 20–50 | 18–30 |
| Far-трикл, cpu мкс/оп | 142 | 119–300 (шумно, ~+30 в среднем) |

Регресс/корректность: `make ext-test` зелёный, `make test` — 419 тестов зелёные.

> **⚠️ Под нагрузкой — только частичная проверка (машина шумит).**
> `bench-socket-throughput` (насыщенный: 50 conn × 2000 ping round-trips), лучшее из
> N, spin=8 vs spin=0 (baseline, тот же билд с `waitSpinIterations=0`):
>
> | билд | best rt/s | разброс прогонов |
> | --- | ---: | --- |
> | baseline (spin=0) | ~165k | 49k–165k |
> | Gosched-8 | ~184k | 81k–184k |
>
> Разброс 3–4× (посторонние контейнеры `pms-*` на хосте) — сигнал не разрешается.
> Читается как **нейтральность спина на насыщенной нагрузке (регресса нет)**, что и
> ожидалось: буфер полон → спин почти не задействуется. `bench-http-load-stats`
> (`/all`, host-wrk) на шумной машине не снимался. **Открытый TODO:** снять
> socket-throughput и `/all` до/после на спокойной машине — там, где трикл-выигрыш и
> отсутствие CPU-регресса подтверждаются числом. Пока `Gosched-8` принят как
> безопасная (без бампа, протокол не тронут, `ext-test`+`test` зелёные, throughput
> без регресса) трикл-оптимизация.

**Честный итог фазы 1.** Спин нейтрален на насыщенной нагрузке (буфер и так не
паркуется — там ноль выигрыша) и даёт частичный выигрыш латентности только в
трикле, где абсолютная латентность и так неважна, ценой небольшого, но реального
CPU на far-трикле. Это **не чистый выигрыш**; по духу репо (идеи закрываются
замером, как B/C) статус — «маргинально, под решение мейнтейнера»: либо принять
`Gosched-8` как дешёвую трикл-оптимизацию (без бампа), либо отклонить и идти в
фазу 2 (shared-memory/eventfd — единственное чистое снятие park/wake без траты CPU)
или фазу 3 (аллокации/GC — прямой адрес жалобы «/all ~1000% CPU»).

## Фаза 2 — shared memory + eventfd вместо блокирующего cgo-`waitAny` (радикально)

Endgame и настоящее «радикальное» решение (идея E из
`php-go-bridge-performance.md`, но с новой мотивацией: доминирует не cgo-*переход*,
а cgo-*парковка*, которую shared memory снимает целиком).

- Go складывает готовые результаты (кадры `buildResultFrame`) в lock-free
  SPSC/MPSC-ring в разделяемой памяти и дёргает **eventfd**.
- PHP ждёт готовности через дешёвый `stream_select`/чтение fd и **забирает
  результаты из ring без cgo-вызова на каждый** — один fd-сигнал может отдать сразу
  несколько готовых кадров (это не батчинг протокола: формат и семантика per-result
  те же, экономится только механизм ожидания).
- Убирает с пути результата и park/wake, и cgo-переход.

**Сложность/риски:** синхронизация ring, владение памятью (Go GC vs
`zend_string`), жизненный цикл буфера, корректность под graceful shutdown. Меняет
транспорт результата → **бамп версии** и согласование с мейнтейнером. Делается
**только** если фаза 0 подтвердила крупную долю park/wake и фазы 1 недостаточно.
Прототип сначала на одной фиче (sleeper) с полным регрессом.

#### Спайк механизма пробуждения (2026-07-24) — гипотеза ПОДТВЕРЖДЕНА

Прежде чем оценивать полный ремонт, изолированно замерили спорную величину: стоимость
пробуждения потребителя, залипшего (а) внутри блокирующего cgo-вызова на Go-канале
(сегодняшний `WaitAny`) против (б) на `eventfd` read в чистом C (механизм фазы 2). В
обоих случаях потребитель — foreign (C-созданная) OS-нить, как PHP-поток; продьюсер —
Go-горутина, как таск; каждый round-trip паркует потребителя (трикл). Стенд:
`ext/cmd/parkwake-spike/main.go`, 300k round-trip, 3 прогона.

| Вариант | wall нс/оп | cpu нс/оп |
| --- | ---: | ---: |
| go-канал пинг-понг (пол, без cgo/foreign-нити) | ~250–360 | ~280–380 |
| **cgo-blocked канал (текущий `WaitAny`)** | **~4900–6100** | **~7200–8300** |
| **eventfd (C-поток, фаза 2)** | **~3300–3700** | **~3200–3500** |

**Вывод: гипотеза подтверждена.** Пробуждение через eventfd дешевле по wall на ~30–40%
и **по CPU в ~2–2.4×** (7.2–8.3 → 3.2–3.5 мкс/пробуждение). Залипание потребителя
*внутри* cgo реально дорогое (needm/dropm foreign-нити + пробуждение cgo-locked M);
eventfd снимает и cgo-вход, и дорогое пробуждение. Оба варианта далеки от go-пола
(~0.3 мкс) — фундамент цены остаётся futex-пробуждение OS-нити, но eventfd-путь заметно
экономнее по CPU.

#### Замер частоты парковок (2026-07-24) — фаза 2 ЗАКРЫТА ПО ДАННЫМ

Спайк показал, что пробуждение дешевле; оставался вопрос — **как часто `WaitAny`
реально паркуется**. Инструментировали `WaitAny` тремя счётчиками (fast = буфер уже
имел результат; spin = поймал спин фазы 1; park = ушёл в блокирующий `select`),
замерили под несколькими профилями конкуренции.

Реальный широкий async DB-веер (`bench-db-lifecycle c=3000 pool=80`, 27000 вызовов
`WaitAny`):

| | fast | spin | park |
| --- | ---: | ---: | ---: |
| DB-веер | 0.0% | **99.9%** | **0.1%** |

Скейлинг по конкуренции (веер 6000 × `usleep(200)`):

| maxConcurrency | fast | spin | **park** |
| ---: | ---: | ---: | ---: |
| ∞ (широкий) | 0% | 100% | **0%** |
| 8 | 0.4% | 91.2% | **8.3%** |
| 2 (узкий) | 0.6% | 55.2% | **44.2%** |

**Вывод — фаза 2 не оправдана.** Спин фазы 1 под высокой конкуренцией/throughput ловит
~99.9%+ результатов — **поток почти не паркуется**, и eventfd-механизму (который дешевле
на пробуждение) оптимизировать нечего. Парк частый **только при низкой конкуренции**
(≤ единиц одновременных операций), где throughput и абсолютная CPU-цена малы, а латентность
упирается в реальный I/O. Именно сценарий «/all ~1000% CPU» — это высокая
конкуренция → парк ~0.1% → фаза 2 его не трогает (тот CPU идёт из Go-драйвера/маршалинга/
zval-churn/спавна горутин, не из park/wake). Многодневный ремонт транспорта ради
пробуждения, которого под нагрузкой почти нет, — не оправдан. **Закрыто по данным.**

Побочно замер **ретроспективно повысил ценность фазы 1**: спин — не просто трикл-твик,
а под реальным веером конвертирует ~99.9% потенциальных парковок в дешёвые спин-поимки.

## Фаза 3 — убрать per-call аллокации → CPU/GC (идея D)

Атакует «/all ~1000% CPU при 3.9k rps против ~410% у /all-native»: на каждый вызов
сейчас одноразовый `Flow` + канал + `context` + spawn горутины + `C.CBytes` malloc +
`zend_string`. Много мусора → давление на Go GC → CPU.

- **Sync-путь инлайн** (пункт роадмапа README «Optimize synchronous-mode
  execution»): вне корутины исполнять хендлер прямо на cgo-нити, без горутины/канала;
  слить `push+wait+stopFlow` в один экспорт `pushWait` (3 cgo → 1). Правит
  `FeatureExecutor::handleSync` (`src/Features/FeatureExecutor.php:112`), lifecycle
  `Flow`, `ext/main.go`.
- **Async-путь:** пул переиспользуемых `Flow`/каналов вместо одноразовых
  (`fresh−reused ≈ 8 аллокаций/вызов` по Go-профилю).

Микробенч single-call этого не видел (ns/op тонет в шуме планировщика), но GC-давление
проявляется именно под серверной нагрузкой — мерить CPU/req на `/all`, а не ns/op.
Меняет протокол (`pushWait`) → бамп версии.

#### Результаты атрибуции (2026-07-24)

Go-бенч аллокаций round-trip'а `Push→WaitAny` (`CGO_ENABLED=0`, sleeper ms=0, чтобы
мерить машинерию, а не таймер; `ext/internal/handler/handler_bench_test.go`,
`-benchtime=2000x`):

| Бенч | ns/op | B/op | allocs/op |
| --- | ---: | ---: | ---: |
| RoundTripFreshFlow (новый flowKey на вызов) | 5083 | 1462 | 23 |
| RoundTripReusedFlow (один flow на все итерации) | 4261 | 644 | 13 |
| **дельта (одноразовый Flow)** | **~822** | **~818** | **~10** |

Подтверждает цель: одноразовый `Flow` (`context.WithCancel` + map `activeTasks` +
struct) добавляет ~10 аллокаций / ~818 B / ~822 нс на вызов. Это ровно **идея D**,
которую профиль `php-go-bridge-performance.md` уже оценил как «< ~8 мкс/вызов,
скромно». На высоком throughput (socket ~150k rt/s) 23→13 allocs/оп режет ~43%
аллокаций → меньше GC → меньше CPU; но абсолютная per-call экономия мала.

**Оценка риска реализации (важно).** Обе ветки фазы 3 — правка ядра конкурентности:

- **Пул Flow/каналов** (без бампа, протокол не тронут) — но переиспользование
  `Flow`-объекта опасно при буферизованном канале результатов: «протухший» результат
  старого flowKey может ещё лежать в буфере, пока объект переиспользован под новый key
  → мисроутинг. Требует аккуратной эпохи/поколения flow и полного регресса на
  конкурентности.
- **Инлайн sync + `pushWait`** — меняет экспорты → **бамп версии + согласование с
  мейнтейнером** (по правилам репо).

Вывод: цель подтверждена, но выигрыш модест (= идея D), а цена — инвазивная правка
ядра с риском по конкурентности и/или бамп. По правилу «нельзя ломать конкурентность»
и культуре измеримых гейтов — (а) стоит ли модест-выигрыш риска, (б) какая ветка первой,
(в) обязателен предварительный замер CPU/req под `socket-throughput`/`/all`, что
аллокации реально двигают CPU под нагрузкой (а не только allocs/op).

#### Реализация — шаг 1: пул Flow (2026-07-24, без бампа)

Сделано (`ext/internal/flows/flows.go`, `flow.go`): `sync.Pool` переиспользует
`Flow`-структуры и их `activeTasks`-map (через `clear()`, не переаллоцируя) между
`DeleteFlow` и следующим `InitFlow`. Одноразовый flow — sync-путь и собственный flow
каждой async-корутины — больше не аллоцирует свежие `Flow` + map на вызов. Остаётся
только per-flow cancel-context (`context.WithCancel` нельзя переиспользовать после
отмены).

**Безопасность.** Flow-ключи глобально уникальны и никогда не переиспользуются
(`uniqid` на sync-пути, монотонные счётчики `sp_N`/`cb_N`), поэтому структура,
переармленная под новый ключ, невидима для «протухших» результатов старого ключа: они
маршрутизируются по своему строковому ключу, которого `GetFlow` уже не знает → дроп до
`OnDelivered`. Доступ к пулу сериализован под `Flows.mutex`; task-горутины
`Flow`-структуру не трогают (держат свой `Task` со своим захваченным контекстом и общий
results-канал). На `Destroy` весь `Flows` c пулом заменяется.

**Замер (детерминированный — шум машины на allocs/op не влияет),
`handler_bench_test.go`, `-benchtime=5000x`:**

| Бенч | B/op до→после | allocs/op до→после |
| --- | ---: | ---: |
| RoundTripFreshFlow | 1462 → **1110** | 23 → **20** |
| RoundTripReusedFlow | 644 → 643 | 13 → 13 (не задет) |

−3 аллокации / −352 B на одноразовый flow (~13% / ~24%). Регресс: `make ext-test`
зелёный, `make test` — 419 тестов зелёные.

#### Шаг 2 — убрать per-flow `context.WithCancel` — ✖ан ОТКЛОНЕНО (корректность)

Идея: tasks деривят от handlerCtx, `Flow.Cancel` явно отменяет активные tasks — сняло
бы ещё ~2–3 аллокации/flow. **Отклонено по анализу кода:** курсорный task после
первого батча удаляется из `activeTasks` (`Flow.OnDelivered`: `delete` всегда, а
`Cancel` не вызывается при `hasNext`), оставаясь «плавающим» — живёт только через свой
контекст, который сейчас отменяется **пропагацией flow-контекста** на `StopFlow`
(`states.Start` вешает `context.AfterFunc` на task-контекст,
`ext/internal/states/states.go:45`). Явная отмена по `activeTasks` в `Flow.Cancel`
**пропустила бы плавающий курсорный task** → его состояние (курсор БД, тело запроса)
утекло бы до `Destroy`. Per-flow контекст — load-bearing механизм очистки курсоров на
стопе флоу; убирать нельзя без отдельного трекинга плавающих tasks (непропорциональная
сложность ради ~2 аллокаций).

#### Шаг 3 — инлайн sync + `pushWait` — ✖ан ОТКЛОНЕНО (размен)

Идея: sync-путь исполняет хендлер инлайн (без горутины) одним экспортом `pushWait`
(3 cgo→1). **Отклонено по размену:** выигрыш — только spawn горутины + rendezvous
(~0.5–1 мкс/вызов по Go-профилю) + пара аллокаций, и только на **sync-пути** (менее
важном, чем async — главная ценность библиотеки). Цена — **бамп версии + правка C-ABI**
(`sconcur.c`, новый экспорт) **+ смена модели исполнения** («всё через планировщик» →
инлайн на cgo-нити). Профиль `php-go-bridge-performance.md` уже закрыл идею C (слияние
cgo ничтожно, 0.082 мкс/переход) и оценил идею D как «< ~8 мкс, инвазивно, только если
единицы мкс стоят правки ядра». <1 мкс на sync-пути за бамп + инвазивную правку — не
стоит. Пункт роадмапа «Optimize synchronous-mode execution» остаётся, но не через это.

## Фаза 4 — считать в Go + ленивый zval-декод (идеи I, H)

Точечно, по фичам, атакует ту часть ~370 мкс/оп, что не «Go-драйвер», а «маршалинг +
сборка zval»:

- **I — считать в Go:** где PHP нужен только итог (count/hash/агрегат/bulk-результат)
  — считать на Go-стороне и вернуть маленький результат, не перекачивая полный набор
  (обобщение паттерна download, `internal/features/httpclient/download.go`).
- **H — ленивый декод результата:** для выборок декодировать поля/строки по
  обращению вместо жадной сборки вложенных массивов на каждый ответ (снижает
  zval-churn — вероятную доминанту на дешёвых операциях). Частично уже есть через
  `next()`-стриминг.

Не трогает протокол ядра (меняются payload'ы/фичи) — версия по необходимости.

#### Статус — ○ ОТЛОЖЕНО (вне рамки этой темы)

Фаза 4 — не про park/wake и не про per-call аллокации ядра, а про стоимость данных
конкретной фичи (маршалинг + сборка zval пропорционально размеру результата). Это
**ортогональная, пофичевая** работа: каждый пункт — отдельный дизайн на своей фиче
(«считать count/агрегат в Go», «ленивый декод строк выборки»), со своим замером на
payload-свипе (см. [`benchmarks-roadmap.ru.md`](benchmarks-roadmap.ru.md), «Свип размера
payload») и своим регрессом. Обобщённого выигрыша «в ядре» тут нет — идея A уже убрала
двойную сериализацию/декод конверта, а дальше payload нужен вызывающему как есть.
Правильное место — отдельные пофичевые задачи под конкретный доказанный горячий кейс
(большие выборки), а не эта тема координации границы. В рамках park/wake-темы — вне
рамки.

## Порядок и гейты

1. **Фаза 0** — атрибуция (обязательный гейт).
2. **Фаза 1** — spin-before-park (дёшево, без бампа; делаем сразу, это же и есть
   проверка гипотезы park/wake числом «до/после»).
3. **Фаза 2** — shared memory/eventfd, только если 0 подтвердила крупную долю
   park/wake и 1 недостаточно (бамп, согласование).
4. **Фаза 3** — CPU/аллокации, параллельно (бамп на `pushWait`).
5. **Фаза 4** — точечно по фичам.

Каждая фаза с изменением ядра/протокола — по правилам проекта: бамп версии
расширения (один раз на ветку), согласование с мейнтейнером, полный регресс.

## Как измерять

- Скейлинг park/wake: `Sleeper::usleep(1)`, `maxConcurrency: 0`,
  N=1000/2000/4000/8000, per-op wall+CPU до/после.
- `make bench-db-lifecycle c=10000` — async vs native/sync.
- `make bench-all` — регресс по фичам (async не должен просесть нигде).
- `bench-http-load-stats` (`/all`) — rps, p50/p99, CPU/req.
- Лучшее/медиана из N прогонов на спокойной машине (троттлинг даёт разброс до 4×).
