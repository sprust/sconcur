# Небатчинговая оптимизация координации веера: цена park/wake на `waitAny`

> **Статус: предложение (2026-07-24), в работе с фазы 1.** Небатчинговая
> альтернатива плану [`php-go-boundary-batching.md`](php-go-boundary-batching.md):
> та же адресуемая статья (⅔ потолка throughput, теряемые на round-trip'ах
> `waitAny` под конкуренцией), но лечится **не** уменьшением числа переходов
> (батч), а **устранением стоимости парковки/пробуждения** PHP-потока на каждом
> переходе. Опирается на профили [`php-go-bridge-performance.md`](php-go-bridge-performance.md)
> и [`async-fan-out-optimization.ru.md`](async-fan-out-optimization.ru.md).
> Батчинг по условию задачи не рассматриваем.

## Мотивация: неатрибутированный разрыв

Профили закрыли «glue-идеи» (cgo-переход `0.082 мкс`, `msgpack pack/unpack
≈ 0.1–0.2 мкс`, framing результата — идея A реализована). Но в них же осталась
несостыковка, которая и есть цель этого плана:

- **Координация веера ≈ 130 мкс CPU на приостанавливающуюся корутину**
  (`suspend → waitAny → resume`), при сумме примитивов ~2 мкс — разрыв **~128 мкс
  на корутину не атрибутирован**.
- **Пустой веер (корутины не паркуются) — ~11 мкс/корутина**, реальный веер —
  ~130. Разница ≈ прямая цена **парковки PHP-потока и его пробуждения через
  границу**.
- **Потолок throughput `/all`:** CPU/req даёт расчётные ~7900 rps, измерено
  **~2580 (~⅓)**; авторы профиля списывают остаток на «round-trip'ы
  `waitAny`/планировщика под конкуренцией». Это единственная крупная статья, не
  разложенная до конца.

**Ключевая гипотеза.** Прошлые микробенчи промахнулись измерить именно эту статью:
они мерили либо **неблокирующий** cgo (`tasksCount`, 0.082 мкс), либо изолированное
рандеву при `CGO_ENABLED=0` (~0.5 мкс) — то есть **без реального парка cgo-нити и её
пробуждения Go-планировщиком с другого OS-потока**. В проде же `waitAny` —
блокирующий cgo `select` на канале (`ext/internal/handler/handler.go:62`): при
пустом буфере PHP-поток паркуется на futex, и завершившаяся горутина БД должна
разбудить cgo-locked поток через границу. Эти park/wake round-trip'ы и съедают ⅔
потолка.

**Корень.** PHP-сторона строго однопоточна и сериализует все результаты через одну
блокирующую точку `waitAny`. Go параллелит на N ядер, а результаты втягиваются по
одному с парковкой. Батчинг режет *число* переходов; этот план режет *стоимость
парковки на каждом переходе* — ортогональная и нетронутая ось.

**Оговорка о свежести цифр.** После профиля 2026-07-02 канал результатов стал
буферизованным (`resultsBufferSize = 1024`, `ext/internal/handler/handler.go:23`),
чтобы убрать rendezvous «две futex-побудки на результат». Поэтому цифры 130/⅓ могли
измениться, и план начинается с обязательной атрибуции (фаза 0), гейтящей всё
остальное — как замеры раньше закрыли идеи B/C.

## Цель

1. Атрибутировать разрыв ~128 мкс/корутина числом на сегодняшнем (буферизованном)
   коде.
2. Снять цену park/wake на горячем пути `waitAny` — без изменения протокола там, где
   это возможно (фаза 1), и радикально через shared memory (фаза 2).
3. Срезать per-call аллокации → давление на Go GC → CPU под серверной нагрузкой
   (фаза 3).
4. Точечно убрать перекачку данных через границу, где PHP нужен только итог
   (фаза 4).

## Честная рамка

- На in-memory single-row операциях паритет с native правками координации не
  гарантирован: серийная PHP-цена операции (pack → переход → resume → unpack →
  сборка zval) сопоставима со стоимостью всей нативной операции. Реальная цель —
  поднять потолок throughput под конкуренцией (закрыть ⅔-разрыв) и снизить CPU, а не
  победить native на голом `SELECT 1`.
- Выигрыш растёт с шириной веера и числом одновременных корутин: чем больше park/wake
  событий в секунду, тем больше отдача.
- Каждая фаза — отдельный измеримый шаг с замером «до/после» (лучшее/медиана из N
  прогонов) и регрессом по всем фичам.

## Фаза 0 — атрибуция park/wake (гейт для всего остального)

Замерить сегодняшнюю цену одного результата через `waitAny`, разложив на:

1. **Go-сторона, пустой vs полный буфер.** Go-бенч (`CGO_ENABLED=0`,
   `ext/internal/handler`) `Handler.WaitAny`: (а) буфер заполнен — быстрый
   неблокирующий receive; (б) буфер пуст — реальная парковка на канале + пробуждение
   горутиной-продьюсером с другого P. Мерить ns/op и futex-побудки
   (`perf stat -e ...` или `GODEBUG=schedtrace`).
2. **Реальный cgo park/wake из PHP.** Микробенч: веер `Sleeper::usleep(1)` такой
   ширины, что PHP-поток стабильно опустошает буфер и паркуется; wall+CPU через
   `getrusage` на один round-trip. Сравнить с тем же веером, но заранее завершёнными
   задачами (буфер всегда полон — парка нет). Дельта = чистая цена park/wake.
3. **Разбивка PHP-стороны на результат:** `parseWaitResponse` + `TaskResultDto`
   (`src/Connection/Extension.php:201`) отдельно от `resumeByResult`
   (`src/Scheduler/Scheduler.php:460`).

**Критерий гейта.** Фазы 1–2 делаются, только если цена park/wake (п.1–2) —
заметная доля round-trip'а под конкуренцией (ориентир ≥ ~10%). Иначе закрыть их как
закрыты B/C и дополнить карту в `php-go-bridge-performance.md`.

**Инструмент.** Готовые харнессы: `make bench-db-lifecycle`, `bench-all`,
`bench-http-load-stats` (`/all`), плюс профильный скейлинг `Sleeper::usleep(1)` по
N из `async-fan-out-optimization.ru.md`.

## Фаза 1 — spin-before-park в `waitAny` (без протокола, без бампа)

Самый дешёвый рычаг: под веерной нагрузкой результат приходит в течение
микросекунд, но `waitAny` при пустом буфере сразу паркует поток. Перед блокирующим
`select` крутить короткий **адаптивный спин** неблокирующих `select { case
<-results: default: }` (несколько итераций с `runtime.Gosched()`), и только потом
падать в блокирующий `select`. Если результат успевает прийти в окне спина — парка и
futex-пробуждения нет вовсе.

- Правка — только `ext/internal/handler/handler.go` (`WaitAny`); протокол,
  экспортируемые функции и PHP-сторона не меняются → **бампа версии нет**.
- Спин — только в `WaitAny` (горячий путь планировщика), **не** в `WaitAnyTimeout`
  (это idle-poll серверного цикла на 250 мс — спин там жёг бы CPU на простое).
- Число итераций спина — тюнируемая константа `waitSpinIterations`; подобрать по
  фазе 0 (баланс «CPU спина» vs «цена парковки»). Начать с малого (напр. 30–100
  итераций с Gosched) и мерить.
- Устаревшие результаты (`deliver()` вернул false из-за остановленного flow) в
  спине дренируются так же, как в блокирующем цикле.

**Риск:** спин — это CPU в обмен на латентность. Если результаты приходят *реже*
окна спина (узкий веер, редкие события), спин жжёт CPU впустую. Поэтому: (а) спин
короткий; (б) замер CPU обязателен — метрика успеха включает «CPU не вырос на
редких/idle сценариях». Возможен адаптивный спин (backoff по недавней истории
попаданий), если фиксированный даёт регресс CPU на узких веерах.

**Критерий успеха:** rps `/all` и `socket-throughput` растут, p50 падает, CPU/req не
растёт (в идеале падает за счёт снятых futex-побудок); `bench-all` без регресса.

## Фаза 2 — shared memory + eventfd вместо блокирующего cgo-`waitAny` (радикально)

Endgame и настоящее «радикальное» решение (идея E из
`php-go-bridge-performance.md`, но с новой мотивацией: доминирует не cgo-*переход*,
а cgo-*парковка*, которую shared memory снимает целиком).

- Go складывает готовые результаты (кадры `buildResultFrame`) в lock-free
  SPSC/MPSC-ring в разделяемой памяти и дёргает **eventfd**.
- PHP ждёт готовности через дешёвый `stream_select`/чтение fd и **забирает
  результаты из ring без cgo-вызова на каждый** — один fd-сигнал может отдать сразу
  несколько готовых кадров (это не батчинг протокола: формат и семантика per-result
  те же, экономится только механизм ожидания).
- Убирает с пути результата и park/wake, и cgo-переход.

**Сложность/риски:** синхронизация ring, владение памятью (Go GC vs
`zend_string`), жизненный цикл буфера, корректность под graceful shutdown. Меняет
транспорт результата → **бамп версии** и согласование с мейнтейнером. Делается
**только** если фаза 0 подтвердила крупную долю park/wake и фазы 1 недостаточно.
Прототип сначала на одной фиче (sleeper) с полным регрессом.

## Фаза 3 — убрать per-call аллокации → CPU/GC (идея D)

Атакует «/all ~1000% CPU при 3.9k rps против ~410% у /all-native»: на каждый вызов
сейчас одноразовый `Flow` + канал + `context` + spawn горутины + `C.CBytes` malloc +
`zend_string`. Много мусора → давление на Go GC → CPU.

- **Sync-путь инлайн** (пункт роадмапа README «Optimize synchronous-mode
  execution»): вне корутины исполнять хендлер прямо на cgo-нити, без горутины/канала;
  слить `push+wait+stopFlow` в один экспорт `pushWait` (3 cgo → 1). Правит
  `FeatureExecutor::handleSync` (`src/Features/FeatureExecutor.php:112`), lifecycle
  `Flow`, `ext/main.go`.
- **Async-путь:** пул переиспользуемых `Flow`/каналов вместо одноразовых
  (`fresh−reused ≈ 8 аллокаций/вызов` по Go-профилю).

Микробенч single-call этого не видел (ns/op тонет в шуме планировщика), но GC-давление
проявляется именно под серверной нагрузкой — мерить CPU/req на `/all`, а не ns/op.
Меняет протокол (`pushWait`) → бамп версии.

## Фаза 4 — считать в Go + ленивый zval-декод (идеи I, H)

Точечно, по фичам, атакует ту часть ~370 мкс/оп, что не «Go-драйвер», а «маршалинг +
сборка zval»:

- **I — считать в Go:** где PHP нужен только итог (count/hash/агрегат/bulk-результат)
  — считать на Go-стороне и вернуть маленький результат, не перекачивая полный набор
  (обобщение паттерна download, `internal/features/httpclient/download.go`).
- **H — ленивый декод результата:** для выборок декодировать поля/строки по
  обращению вместо жадной сборки вложенных массивов на каждый ответ (снижает
  zval-churn — вероятную доминанту на дешёвых операциях). Частично уже есть через
  `next()`-стриминг.

Не трогает протокол ядра (меняются payload'ы/фичи) — версия по необходимости.

## Порядок и гейты

1. **Фаза 0** — атрибуция (обязательный гейт).
2. **Фаза 1** — spin-before-park (дёшево, без бампа; делаем сразу, это же и есть
   проверка гипотезы park/wake числом «до/после»).
3. **Фаза 2** — shared memory/eventfd, только если 0 подтвердила крупную долю
   park/wake и 1 недостаточно (бамп, согласование).
4. **Фаза 3** — CPU/аллокации, параллельно (бамп на `pushWait`).
5. **Фаза 4** — точечно по фичам.

Каждая фаза с изменением ядра/протокола — по правилам проекта: бамп версии
расширения (один раз на ветку), согласование с мейнтейнером, полный регресс.

## Как измерять

- Скейлинг park/wake: `Sleeper::usleep(1)`, `maxConcurrency: 0`,
  N=1000/2000/4000/8000, per-op wall+CPU до/после.
- `make bench-db-lifecycle c=10000` — async vs native/sync.
- `make bench-all` — регресс по фичам (async не должен просесть нигде).
- `bench-http-load-stats` (`/all`) — rps, p50/p99, CPU/req.
- Лучшее/медиана из N прогонов на спокойной машине (троттлинг даёт разброс до 4×).
