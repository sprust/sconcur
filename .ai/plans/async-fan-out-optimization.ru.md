# Оптимизация веера async и снятие лимита корутин

Статус: пункты 1 (suspend-first push, дефолт `maxConcurrency = 0`), 2 (буфер
results-канала) и 5 (счётчик вместо `uniqid`) реализованы и подтверждены
замерами; пункты 3–4 (батчинг, требует бампа minor) — под вопросом, решение за
мейнтейнером: после пунктов 1+2 координационная цена веера уже сильно упала, а
батчинг — самая инвазивная часть (изменение протокола). Продолжение тем
[`wait-group-max-concurrency.md`](wait-group-max-concurrency.md) (квадрат O(N²)) и
[`php-go-bridge-performance.md`](php-go-bridge-performance.md) (профили налога границы).

## Цель

1. Убрать квадратичную деградацию веера и вернуть дефолт `maxConcurrency = 0`
   (безлимит); сам параметр остаётся как опциональный backpressure.
2. Сократить разрыв `async` с `native` на in-memory операциях и увеличить отрыв там,
   где async уже выигрывает (колонки с задержкой, клиенты, `/all`).

## Честная рамка

На in-memory single-row SQL (`SELECT 1` за ~27 мкс у native PDO) async native не
победит правками обвязки: серийная PHP-цена одной операции (pack → переход → resume →
unpack → сборка zval) больше, чем стоит вся операция у native. Профиль в
[`php-go-bridge-performance.md`](php-go-bridge-performance.md) это закрыл: glue-запас —
единицы мкс. Реалистичная цель — паритет или близко к нему на in-memory и снятие
лимита; выигрыш растёт с реальной задержкой I/O.

Отправные цифры:

- координация веера ≈ 130 мкс CPU на корутину (suspend → `waitAny` → resume) — на два
  порядка дороже примитивов (fiber switch ~1 мкс, cgo 0.082 мкс, unpack 0.17 мкс);
- квадрат O(N²): N=10000 у async ≈ 33 с против 3–5 с у sync
  (`tests/benchmarks/db-lifecycle.php`); изоляция показала — квадрат появляется только
  у фиберов, пересёкших cgo-границу и оставшихся живыми;
- в таблицах `docs/benchmarks.ru.md` `count=100`, поэтому лимит 256 на опубликованные
  цифры не влияет — он бьёт только по большим N.

## Пункты

### 1. Suspend-first push: cgo не вызывается со стека фибера (в работе)

Сейчас корутина сама зовёт `Extension::push()` изнутри фибера
(`src/Features/FeatureExecutor.php`), потом suspend'ится. Гипотеза квадрата —
реакция Go-рантайма на cgo-вызовы с множества разных живых чужих стеков (в Go есть
`callbackUpdateSystemStack` для вызовов с «неизвестных» стеков; точный механизм не
установлен, но изоляционные бенчи однозначно указывают на пересечение границы со
стека фибера).

Правка (чистый PHP, протокол не меняется, бампа версии нет):

- async-путь `FeatureExecutor::exec()`/`next()` не пушит, а сразу
  `Fiber::suspend(new PendingPushDto(...))` / `PendingNextDto`;
- сторона, к которой вернулось управление (`WaitGroup::launch` после `start()`,
  `Scheduler::spawn`, `Scheduler::resumeCoroutine` после `resume()`), выполняет
  push/next через `Scheduler::dispatchPendingTask()` со своего стека и регистрирует
  `State::addFiberTask`;
- ошибка push бросается обратно в корутину через `Fiber::throw()` в точку suspend —
  там она, как и раньше, оборачивается в `TaskExecutionException`; корутина может
  поймать её и сделать следующий async-вызов (диспетчер обрабатывает в цикле);
- sync-путь не меняется.

Нюансы:

- при вложенных группах `add()` выполняется на стеке родительской корутины — push
  идёт с одного её стека, а не с N дочерних; резюмы всегда идут с корневого стека
  планировщика;
- проверка `UnexpectedTaskKeyException` на async-пути уходит (фибер больше не знает
  свой taskKey до результата); маршрутизация гарантируется маппингом
  `State::pullFiberByTask`, регистрируемым в момент push;
- best-effort-очистка в `WaitGroup::stop()`: если finally корутины при unwinding
  делает новый async-вызов, задача теперь не уходит в Go вовсе (раньше уходила и
  осиротевала — flow всё равно удалялся).

Критерий успеха: per-op не растёт с N на скейлинге `Sleeper::usleep(1)`
(N=1000/2000/4000/8000, `maxConcurrency: 0`); db-lifecycle async при N=10000
приближается к sync. После подтверждения — дефолт `maxConcurrency = 0`.

#### Результаты (2026-07-04, i7-12700F, контейнер)

Гипотеза подтверждена: квадрат давали cgo-вызовы со стеков фиберов. Скейлинг
`Sleeper::usleep(1)`, `maxConcurrency: 0`, свежий процесс на каждый N, per-op мкс
(до = ветка на коммите `29b10ca`, после = suspend-first):

| N | до, мкс/оп | после, мкс/оп |
| ---: | ---: | ---: |
| 1000 | 946 | 21.7 |
| 2000 | 1612 | 24.1 |
| 4000 | 2824 | 24.6 |
| 8000 | 5380 | 22.6 |

Per-op стал константой (линейный веер); на N=8000 — ~240× быстрее. Дефолт
`maxConcurrency` переведён в 0 (безлимит), параметр остался как backpressure.

`make bench-db-lifecycle c=10000 runs=1 pool=80` (in-memory, безлимитный веер,
кап — пул 80 соединений на Go-стороне), мс на фазу из 10000 операций:

| БД | фаза | native | sync | async |
| --- | --- | ---: | ---: | ---: |
| mysql | read | 379 | 3932 | 621 |
| mysql | delete | 1418 | 4178 | 457 |
| pgsql | read | 507 | 2349 | 647 |
| pgsql | delete | 511 | 2813 | 512 |
| mongodb | read | 1095 | 4268 | 521 |
| mongodb | delete | 886 | 3524 | 456 |

async обгоняет native на mongodb (обе фазы, ~2×) и mysql delete (~3×), паритет на
pgsql delete; проигрывает только на дешёвых read (~1.3–1.6×). Раньше async на
N=10000 был ~33 с/фазу — хуже всех режимов на порядок.

Нюанс окружения: `pool=100` (дефолт бенча) валит pgsql-фазу ошибкой
`FATAL: sorry, too many clients already` — у Postgres в docker-compose
`max_connections=100`, и быстрый async теперь реально насыщает пул (плюс
native-PDO-соединение того же процесса). Запускать с `pool` < `max_connections`
или поднять лимит сервера.

### 2. Буферизовать общий канал результатов в Go (сделано)

`h.results = make(chan *dto.Result)` был небуферизованным: каждый результат —
rendezvous двух парковок (горутина на send, PHP-поток в recv) и двух
futex-пробуждений. Теперь канал буферизован (`resultsBufferSize = 1024`,
`ext/internal/handler/handler.go`): готовые результаты лежат в канале, `waitAny`
забирает их по fast-path, горутины завершаются не вися на send; за пределами
буфера действует прежний backpressure. Протокол не меняется, порядок FIFO
сохраняется.

Обязательное следствие буфера: результат может лечь в канал, а его flow — быть
остановлен (`StopFlow`) до того, как PHP его заберёт. Такие «протухшие» результаты
раньше физически не доезжали (rendezvous + отмена контекста), теперь — дропаются
на приёме: `deliver()` возвращает false для неизвестного flow, циклы
`WaitAny`/`WaitAnyTimeout`/`Wait` пропускают такой результат и ждут дальше;
`StopFlow` дополнительно вычищает `pending`-корзину flow.

Результат (db-lifecycle N=10000, pool=80, вместе с п.5): async быстрее ещё на
~15–25% по всем трём БД — mysql delete 457 → 341 мс, mysql read 621 → 523,
pgsql delete 512 → 408, mongodb read 521 → 393, mongodb delete 456 → 354.
Чистый веер `Sleeper::usleep(1)` остался линейным (~19–24 мкс/оп): там доминирует
серийная PHP-цена операции, выигрыш rendezvous виден на реальных горутинах БД.

### 3. Батч-драйн результатов: `waitAnyMany` (под вопросом)

Один cgo-вызов возвращает все готовые результаты (кадр из K фреймов). Планировщик
резюмит фиберы из локальной очереди без похода в Go на каждый. Для веера из 100
операций ~100 переходов+пробуждений превращаются в единицы. Изменение протокола →
бамп minor (первый на ветке).

### 4. Батч отправки: `pushMany` (под вопросом)

Планировщик копит payload'ы приостановившихся корутин (после п.1 они и так у него)
и отправляет пачкой одним переходом. Идея F из
[`php-go-bridge-performance.md`](php-go-bridge-performance.md); в паре с п.3 сводит
цену границы веера почти к константе (~515 мкс координации на `/all`).

### 5. `uniqid()` → счётчик в `WaitGroup::add()` (сделано)

`callbackKey` теперь `'cb_' . ++static::$callbackKeyCounter` (монотонный счётчик
по образцу `Scheduler::$spawnCounter`) вместо `uniqid(more_entropy: true)` на
каждый `add()`.

## Фаза 2: async быстрее native на in-memory (ресёрч 2026-07-04)

Вопрос: почему на in-memory дешёвых операциях async всё ещё проигрывает native
(mysql selectOne 40.4 vs 3.7 мс на 100 вызовов и т.п.), если пол границы уже
~22 мкс/оп (веер `Sleeper::usleep(1)`), то есть ниже цены нативной операции?

Эксперименты (контейнер, микробенчи на живых БД; веер = 100 операций):

| Замер | мкс/оп |
| --- | ---: |
| native PDO mysql, prepared execute+fetch | 38.0 |
| sconcur sync mysql fetchAll (тёплое 1 соединение) | 266 |
| async веер mysql, холодный пул | 470 |
| async веер mysql, частично прогретый пул (2–3-й веер) | 150–157 |
| async веер mysql, полностью прогретый пул (5–6-й веер) | **27–46** |
| native pgsql (из таблиц доки) | ~54 |
| async веер pgsql, холодный пул | 970 |
| async веер pgsql, тёплый пул (дефолтный stmt-cache pgx) | **62** |
| async веер pgsql, `simple_protocol` | 184 (хуже — не брать) |
| native mongo findOne / insertOne | 111 / 57 |
| async веер mongo findOne холодный / тёплый | 503 / **71** |
| async веер mongo insertOne тёплый | **47** |

Выводы — «async медленнее native на in-memory» в текущих таблицах почти целиком
артефакт методики и настроек, а не архитектуры:

1. Холодный пул. `Benchmarker` не греет: sync-фаза держит 1 соединение, а
   async-веер открывает до `maxOpenConns` соединений прямо в замере (плюс у Go
   `database/sql` дефолт MaxIdleConns=2 — между веерами пул схлопывается и
   хендшейки повторяются). Цена — 3–15× (pg: 970 → 62 мкс).
2. MySQL и round-trip'ы: гипотеза про Prepare+Execute+Close не подтвердилась —
   фасад `Mysql\Connection` уже добавляет `interpolateParams=true` в DSN по
   умолчанию (`withInterpolateParams`), и пулы в Go ключуются по DSN, поэтому
   в эксперименте «plain» и «interp» соединения делили один интерполированный
   пул. Наблюдавшаяся разница 150 → 27 мкс между веерами — продолжение
   прогрева (пул, кэши сервера), а не флаг. По wire-протоколу mysql-путь уже
   в паритете с PDO (1 round-trip).
3. PostgreSQL: дефолтный режим pgx (кэш prepared statement) уже оптимален,
   `simple_protocol` хуже.
4. Mongo: тот же эффект прогрева пула Go-драйвера.

На тёплых пулах async веером обгоняет native уже на in-memory: mysql 27–46 vs
38, mongo findOne 71 vs 111, insertOne 47 vs 57; pgsql — паритет (62 vs ~54).

### Шаги фазы 2 (скорректированы после проверки кода)

1. `Benchmarker`: фаза прогрева перед замером — несколько sync-операций + один
   отбрасываемый async-веер полного размера (сделано; `warmup: false` у
   `mongodb-createIndex` — лимит 64 индексов на коллекцию). Это честно: native
   тоже входит в замер с готовым соединением и prepared statement.
2. Библиотека: если задан `maxOpenConns`, а `maxIdleConns` нет — дефолт
   `maxIdleConns = maxOpenConns` (иначе Go держит 2 idle и пул схлопывается
   между веерами). Чистый PHP (`Sql\Connection`), протокол не меняется
   (сделано).
3. `interpolateParams` — ничего делать не нужно: уже дефолт фасада
   `Mysql\Connection`, задокументирован в `docs/mysql.ru.md`.
4. Mongo-бенчи: прогрев закрывается пунктом 1 (отбрасываемый веер прогревает
   пул драйвера до ширины веера).
5. Переснять таблицы БД: ожидание — async ≥ native на большинстве in-memory
   строк. Сделано: async обгоняет native в 13 строках (mongo insertOne 14.1 vs
   19.3, insertMany ×4.4, count/bulkWrite/updateMany ~×7, createIndex; mysql
   insert 4.0 vs 7.2, delete 5.5 vs 7.0; оба клиентских ~×40), на остальных
   разрыв упал с ~10× до ~1.5–2× (mysql selectOne 5.2 vs 3.5 — было 40.4).
6. `/all`: повторный wrk-замер — кап пула теперь держит 5 idle-соединений
   (пункт 2). Сделано: 3 190 → 3 884 rps (+22%), p50 72 → 63 мс; RoadRunner
   впереди ~1.5× (было ~1.9×).

## Фаза 3: ресёрч «фичи vs native» по db-lifecycle (2026-07-04)

Главная метрика (по решению мейнтейнера) — `make bench-db-lifecycle c=100`:
изолирует стек фич от серверного контекста. Пустая ручка и `/all-native`
доказали, что сервер/транспорт не уступают нативному стеку — вся дельта в фичах.

Сделано и намерено:

1. Прогрев db-lifecycle выровнен с Benchmarker: async-режим греется веером
   полной ширины (было `min(10, count)` — веер из 100 раскручивал пул с 10 до
   80 соединений внутри замера). После фикса (c=100, медиана из 3):
   mysql — async выигрывает обе фазы (read 5.1 vs 5.3, delete 4.7 vs 9.7);
   pgsql — read выигрывает (4.0 vs 6.0), delete проигрывает (10.4 vs 5.8, шумно);
   mongo — проигрывает (read 14.3 vs 7.2, delete 11.3 vs 9.3).
2. Interleaved A/B mongo findOne (один процесс, 9 повторов): fan best 59.2 vs
   native best 92.1 мкс/оп — в чистых окнах веер быстрее; медианы (251 vs 188)
   тонут в шуме — на машине разброс 4× между повторами одного и того же замера
   (ноутбук, троттлинг: `CPU scaling 17%` в lscpu). Вывод: mongo-строки
   db-lifecycle страдают от шума и динамики пула, а не от дефекта фичи.
3. Гипотеза «поднять maxConnecting у Go-драйвера mongo» проверена и отвергнута:
   `maxConnecting=64` в URI стабильно хуже дефолтных 2 (шторм одновременных
   SCRAM-хендшейков; cold fan 293–602 vs 111–141 мкс/оп). Ничего не меняем.
4. Покомпонентный пол PHP-стороны (best-of-7, один процесс): noop-корутина
   7.5 мкс/оп; sleeper-веер 16.3; mysql exec веером 22.8; fetchAll по id 24.7 —
   против 34.2 у native PDO. Go-сторона целиком (Push→database/sql→WaitAny,
   веер 100) — 25 мкс/оп wall. Явного «жирного» дефекта в фичах не осталось;
   остаточные резервы — микро (пол корутины, ~15 мкс границы на операцию),
   их могли бы срезать батчи (пп. 3–4), но выигрыш скромный.

Открытое: CPU-цена стека под серверной нагрузкой (`/all` ~1000% CPU при
3.9k rps против ~410% у `/all-native`) — для атрибуции нужен профиль
(perf на хосте закрыт `perf_event_paranoid=4`; pprof в расширении мейнтейнер
отклонил). Достоверные медианы db-lifecycle требуют спокойной машины
(без троттлинга) — текущие прогоны дают разброс до 4×.

## Фаза 4: дисковые тома вместо tmpfs (2026-07-04)

По решению мейнтейнера БД переведены на именованные дисковые тома
(`make bench-reset` сбрасывает состояние), все бенчи пересняты на спокойной
машине (разброс wrk < 1%). Итоги:

- db-lifecycle (c=100, медиана из 5): async обгоняет native на mysql read
  (14.5 vs 25.5 мс), mysql delete (43.9 vs 652 — ~15×), pgsql delete
  (14.6 vs 126 — ~8.6×), mongo read (18.2 vs 23.5); паритет pgsql read;
  проигрывает только mongo delete (13.6 vs 7.6).
- Таблицы: async быстрее native в 14 строках, включая mysql insert ~12× и
  transaction ~9× (fsync накладывается веером).
- wrk `/all`: SConcur ≈2 680 rps против RoadRunner ≈446 — веер быстрее ~6×
  (у последовательного воркера fsync'и складываются, p50 0.6 c);
  `/all-native` ≈451 подтверждает паритет серверного слоя. Цель
  «bench-http-load-stats лучше bench-rr-load-stats» достигнута на
  реалистичных условиях.

1. Этап 1 (без изменения протокола, без бампа): п.1 + п.2 + п.5 — выполнен,

2. Этап 2 (протокол, один minor-бамп на ветку): п.3 + п.4 — под вопросом; если

## Ожидаемый эффект

- db-lifecycle N=10000: async с ~33 с до уровня sync и ниже, без лимита.
- Async-колонки in-memory: снижение в разы за счёт п.2–4 (паритет с native на
  single-row SQL не гарантируется — см. рамку).
- Колонки с задержкой и клиенты: отрыв async растёт (меньше серийной цены на
  операцию).
- `/all` vs RoadRunner: разрыв сузится за счёт координации, на in-memory полностью
  не закроется.

## Как измерять

- Скейлинг квадрата: `Sleeper::usleep(1)`, `maxConcurrency: 0`,
  N=1000/2000/4000/8000, per-op до/после.
- `make bench-db-lifecycle c=10000` — async vs sync.
- `make bench-all` — регресс по фичам (async не должен просесть нигде).
- Медиана/лучшее из 3 прогонов, как в методологии доки.
