# Лимит параллелизма группы — `WaitGroup::create(maxConcurrency: N)`

Статус: план. Краткий пункт — в [README → Планы](../../README.md#планы).

## Зачем

Сейчас `add()` сразу прогоняет корутину до первого `suspend` и пушит задачу в Go.
Добавление тысяч задач разом отправляет тысячи задач в Go одновременно — пик по
памяти, числу in-flight Go-задач, открытым соединениям и курсорам. Нужен
backpressure: ограничить число одновременно «в полёте» корутин (запущены, но ещё
не завершены).

По умолчанию `maxConcurrency = 0` — без лимита, текущее поведение (обратная
совместимость).

## API

```php
$wg = WaitGroup::create(maxConcurrency: 100);
```

Остальной API (`add`, `iterate`, `waitAll`, `waitResults`, `stop`) не меняется.

## Механика

- **Отложенный запуск.** Тело запуска корутины выносится из `add()` в приватный
  `launch()` (создание `Fiber`, регистрация в `State`/`Scheduler`, `start()` до
  первого `suspend`). `add()`:
  - генерит `callbackKey`;
  - если живых членов `< N` (или `N == 0`) → `launch()` сразу, как сейчас;
  - иначе → кладёт `{callbackKey, callback}` в очередь ожидания группы и **сразу
    возвращает `callbackKey`**. В Go при этом ничего не уходит.

- **Дренаж — только через планировщик (инвариант не ломается).** Слот
  освобождается, когда член завершается или падает (`Scheduler` →
  `completeCoroutine`/`failCoroutine` → `forget`). Там же планировщик просит
  группу заполнить освободившиеся слоты: `fillSlots()` → `launch()` следующих из
  очереди, пока есть слоты. Единственной точкой запуска и резюма остаётся
  планировщик, поэтому корутины по-прежнему не вкладываются друг в друга по стеку
  вызовов.
  - На верхнем уровне очередь дренажится во время `iterate()` / `Scheduler::run()`.
  - Для `add()` **изнутри** корутины при насыщении — кооперативная приостановка
    `Scheduler::awaitSlot()` (по аналогии с `awaitGroup()`), не блокирующая поток.

## Семантика и нюансы

- Лимит считается по живым (незавершённым) членам; синхронная корутина (без
  async-вызова) занимает слот лишь на миг.
- Порядок выдачи `iterate()` не меняется — по готовности.
- `stop()` сбрасывает и очередь ожидания: отложенные корутины не запускаются.
- Вложенные группы независимы: у каждой свой лимит и своя очередь.
- Исключение из `launch()` отложенной корутины всплывает уже в контексте
  планировщика и должно уходить в `markFailure` группы (как сейчас в
  `failCoroutine`), а не из `add()`.

## Открытые вопросы

- **Дедлок при само-зависимости.** Если внутри корутины добавлять в ту же группу
  и ждать её результат при слишком малом `N`, очередь может не разгрестись.
  Минимум — задокументировать; в идеале — детектить (например, когда все живые
  члены ждут слот/группу, а очередь непуста).
- Нужен ли отдельный счётчик in-flight Go-задач против числа членов (член может
  держать несколько последовательных задач через `next` у стриминга).
- Способ менять лимит на лету или это immutable-параметр группы.
