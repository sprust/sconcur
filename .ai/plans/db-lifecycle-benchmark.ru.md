# План: честный lifecycle-бенч по БД (native-сид → read → delete)

> Реализовано: `tests/benchmarks/db-lifecycle.php`, цель `make bench-db-lifecycle`.
> Ниже — актуальный дизайн. Историческая заметка: первая итерация мерила фазу
> `insert` веером; её заменили на **предзаполнение native-драйвером вне замера**,
> чтобы read/delete всех режимов работали с идентичным датасетом.

## Цель

Один объединённый («большой») бенчмарк по трём БД-фичам — MongoDB (только
показательные операции), MySQL, PostgreSQL — который на **свежесозданной таблице/
коллекции с уникальным именем** прогоняет реалистичный жизненный цикл:

1. создать таблицу/коллекцию (уникальное имя на каждый режим/прогон);
2. **предзаполнить `N` строк/документов через native-драйвер — вне замера** (батч
   multi-row INSERT в транзакции для SQL, батч `insertMany` для Mongo; id/`_id` = `1..N`);
3. прочитать все `N` по id — замер;
4. удалить все `N` по id — замер;
5. удалить таблицу/коллекцию (teardown).

read и delete меряются в трёх режимах: `native` (родной драйвер — `PDO` / драйвер
`mongodb/mongodb`), `sync` (SConcur вне `WaitGroup`), `async` (SConcur веером в
`WaitGroup`). Сид всегда native — датасет идентичен для всех режимов. Без
искусственной задержки — чистый in-memory loopback (данные БД в `tmpfs`).

Это «честная база»: воспроизводимый, свободный от кросс-прогонной грязи замер цены
границы PHP↔Go и оверхеда фиберов на масштабе 10000 операций. Задержка (toxiproxy)
в этой итерации намеренно не вводится — гарнесс проектируется так, чтобы её можно
было добавить потом одним измерением (см. [«Задел на задержки»](#задел-на-задержки)).

## Почему это «максимально честно»

Проблема текущих latency-колонок в [docs/benchmarks.ru.md](../../docs/benchmarks.ru.md)
разобрана отдельно: `in-memory` меряли без прокси, а `+Nмс` — через `toxiproxy`,
поэтому прирост смешивал «прокси-хоп» и «задержку». Здесь этой ловушки нет вовсе
(задержки нет), а честность обеспечивается конструкцией:

- **Изоляция состояния по режиму.** У каждого из `native/sync/async` — собственная
  уникально названная таблица/коллекция. Режимы не мешают друг другу, нет остаточных
  строк/индексов от предыдущего режима или прогона.
- **Одинаковая форма данных.** Строка/документ одинаковы во всех режимах; один и тот
  же `N`.
- **Свежая схема каждый прогон.** Нет накопленного bloat, автоинкремент с нуля,
  индексов сверх PK нет.
- **Прогрев вне замера.** Перед таймингом — несколько «холостых» операций на каждом
  соединении/пуле, чтобы из замера ушла цена хендшейка и прогрева пула.
- **Пик RSS сбрасывается на каждой замеряемой фазе** (`memory_reset_peak_usage`).
- **Медиана из нескольких прогонов** (по умолчанию 3, как в текущей методологии).
- **Явный teardown в `finally`.** Все созданные таблицы/коллекции удаляются даже при
  ошибке; уникальные имена исключают коллизии с прерванными прогонами.
- **Честный native-эталон.** `native` — одиночное соединение `PDO` / клиент драйвера,
  операции по очереди (как в остальных бенчах репозитория). Именно с ним сравнивается
  цена SConcur.

Ожидаемый честный результат (совпадёт с выводами доки): на in-memory `native`
выигрывает, `sync` платит цену границы, `async` — цену границы плюс спавн фиберов;
веер здесь не даёт выигрыша, потому что накладывать нечего. Ценность бенча — чистая,
воспроизводимая цифра этого оверхеда на масштабе.

## Форма фаз

Датасет из `N` строк создаётся **native-драйвером до замера** (батч, быстро,
идентичен для всех режимов). Затем меряются две фазы, каждая = `N` отдельных
операций по ключу (сравнимо между режимами, `async` раскладывает веером):

- **read**: `N` точечных чтений по первичному ключу (`WHERE id = k` / `findOne(_id=k)`,
  `k = 1..N`);
- **delete**: `N` точечных удалений по ключу (`WHERE id = k` / `deleteOne(_id=k)`).

`_id` в Mongo — явное целое `1..N` (как id в SQL), поэтому read/delete по ключу
одинаковы для всех трёх БД. Порядок фаз: read → delete (удаление после чтения).

> Почему сид отдельным native-шагом, а не мерянной фазой insert: так read/delete
> всех трёх режимов стартуют с побайтово идентичного датасета, а быстрый батч-сид
> не смешивает «цену наполнения» с замером. Фаза insert как отдельный веерный замер
> убрана.

## Показательные операции MongoDB

Пользователь просил не гонять все операции Mongo. Берём ровно те, что ложатся в
lifecycle и зеркалят SQL:

- insert → `insertOne` (×`N`);
- read → `findOne` по `_id` (×`N`);
- delete → `deleteOne` по `_id` (×`N`).

`bulkWrite`, `updateMany`, `aggregate`, `count`, `createIndex`, `command` — не включаем.

## Затрагиваемые файлы

### Новый оркестратор
- `tests/benchmarks/db-lifecycle.php` — точка входа. Аргументы: `argv[1]` = `N`
  (default 10000), `argv[2]` = число прогонов (default 3), `argv[3]` = размер пула для
  sync/async (default, напр. 100). Прогоняет `feature × phase × mode`, печатает сводные
  таблицы (по фиче: строки — фазы, колонки — `native/sync/async`: wall ms, время на
  операцию, пик RSS МБ), берёт медиану по прогонам.

Оркестратор НЕ переиспользует `Benchmarker::run()` напрямую: тот чередует
`native→sync→async` для одной операции и не подходит для stateful-lifecycle с изоляцией
состояния по режиму. Вместо этого — свой лёгкий хелпер измерения (`measure(Closure): [wallMs, peakMb]`),
и три изолированных прогона lifecycle (по одному на режим, каждый на своей таблице).

### Расширение резолверов (`tests/impl/`)
Сейчас `prepareBenchmarkTable()` работает с фиксированным именем. Добавить методы под
уникальные имена (стиль — вручную, эти файлы вне cs-fixer/phpstan):

- `TestMysqlResolver` / `TestPgsqlResolver`:
  - `createBenchmarkTable(string $tableName): void` — `CREATE TABLE <name> (...)`;
  - `dropBenchmarkTable(string $tableName): void` — `DROP TABLE IF EXISTS <name>`;
  - при необходимости — перегрузка `getConnection`/`getPdo` без изменений (имя таблицы
    приходит параметром в SQL, соединение общее).
- `TestMongodbResolver`:
  - `getDriverCollectionByName(string $name)` и `getSconcurCollectionByName(string $name)`
    (уже есть похожие `getDriver/SconcurTestCollection(string)` — можно переиспользовать);
  - удаление коллекции — через `drop()` (native драйвер) в teardown.

Генерация уникального имени — в оркестраторе: например `bench_lc_<uniqid>_<mode>`
(SQL-safe: только `[a-z0-9_]`).

### makefile
- Новая цель `bench-db-lifecycle` с параметрами `c` (=`N`), опц. `runs`, `pool`:
  ```
  bench-db-lifecycle:
  	$(PHP_EXT) tests/benchmarks/db-lifecycle.php ${c} ${runs} ${pool}
  ```
- В `bench-all` НЕ добавляю (бенч тяжёлый/длинный — 3 БД × 3 фазы × 3 режима × 3 прогона
  × 10000 операций). Оставляю отдельной целью.

### Документация
- В этой итерации доку **не трогаю** (план — dev-артефакт). После снятия цифр отдельной
  задачей можно добавить раздел в `docs/benchmarks.ru.md`. Задержки/latency-колонки — тем
  более позже.

## Детали реализации

### Изолированный lifecycle на режим
Псевдокод одного прогона одной фичи:
```
foreach ($mode in [native, sync, async]) {
    $table = "bench_lc_{$uniq}_{$mode}";
    createTable($table);          // teardown гарантирован в finally
    warmup($mode);                // холостые операции, вне замера

    $insertMs = measure(fn() => runInsert($mode, $table, $N));
    $readMs   = measure(fn() => runRead($mode, $table, $N));
    $deleteMs = measure(fn() => runDelete($mode, $table, $N));

    dropTable($table);
}
```
- `native`: цикл `N` операций по очереди на одном `PDO` / коллекции драйвера.
- `sync`: цикл `N` операций по очереди через `Connection` SConcur (путь `Extension::wait`).
- `async`: `N` операций веером через `WaitGroup` (пул из `pool` соединений на Go-стороне
  ограничивает фактическую конкурентность — это честный кап и он логируется).

### Прогрев
До замеров прогнать ~несколько операций каждого вида на прогреваемом соединении/пуле,
результат отбросить. Цель — убрать из первого замера цену установки соединений и
инициализации пула.

### Память и фиберы
`async` при `N = 10000` спавнит 10000 фиберов (feature `WaitGroup::maxConcurrency`
ещё не реализована — см. `README.md` «Планы»). Это тяжело по памяти, но это честная
цена, её и показываем (пик RSS на фазу). В `_benchmarker.php` уже стоит
`memory_limit = 1024M` — в оркестраторе выставить так же.

### Корректность / безопасность
- Все `DROP`/`drop()` — в `finally`, чтобы уникальные таблицы не копились в `tmpfs`.
- Не звать `exit()` при незавершённых задачах (правило проекта): каждый `WaitGroup`
  довести `waitResults()` до конца.
- Проверять, что каждая фаза реально отработала (`affectedRows`/`insertedId`/найденный
  документ), иначе замер бессмысленен (как «проверка честности» в доке про RoadRunner).

## Задел на задержки

Гарнесс строится так, чтобы позже добавить latency-измерение честно:
- ввести `toxiproxy` в `docker-compose.yml` как прокси перед mysql/pgsql/mongodb;
- роутить **все** режимы (включая `native` и «0 мс») через прокси, задавая toxic
  `0 / 1 / 5 мс` — тогда прокси-хоп постоянен во всех колонках и прирост = чистая
  задержка (устраняет нечестность текущей доки);
- оркестратор параметризовать уровнем задержки (доп. измерение поверх `feature × phase
  × mode`).

Это отдельная будущая итерация, не в этом плане.

## Проверка

1. Собрать окружение: `make up` (БД подняты).
2. Дымовой прогон: `make bench-db-lifecycle c=100 runs=1` — убедиться, что все фичи/фазы/
   режимы отрабатывают, таблицы создаются и удаляются, сводка печатается.
3. Полный прогон: `make bench-db-lifecycle c=10000` — снять цифры.
4. Анализаторы: `tests/benchmarks` и `tests/impl` **вне** scope phpstan/cs-fixer
   (проверено: `phpstan.neon` и `cs-fixer.dist.php` покрывают только `src` + `tests/feature`),
   поэтому `make php-stan`/`make cs-fixer-check` новый код не увидят — стиль держим по
   правилам проекта вручную. `make test` для чистоты можно прогнать (регрессий не ждём).
