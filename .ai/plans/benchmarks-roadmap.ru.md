# План: расширение бенчмарков — наглядность, честность, трушность

Статус: идеи, не начато. Кандидаты на добавление к текущему набору
(`docs/benchmarks.md`, `tests/benchmarks/`); инфраструктура (`_benchmarker.php`,
msleep-эндпоинты серверов) переиспользуется. Связанные планы:
[`db-lifecycle-benchmark.ru.md`](db-lifecycle-benchmark.ru.md) (честная база на
идентичном датасете), [`php-go-bridge-performance.md`](php-go-bridge-performance.md)
(профиль налога границы),
[`async-fan-out-optimization.ru.md`](async-fan-out-optimization.ru.md)
(координационная цена веера).

## Наглядность

1. **Свип ширины веера.** Одна операция (например, pgsql `insert`) при
   N = 1 / 10 / 50 / 100 / 500 / 1000 корутин. Сейчас все цифры при N=100 — не
   видно, где выигрыш насыщается и где упираемся в пул (`maxOpenConns: 50`) или
   планировщик. Отвечает на вопрос читателя «сколько корутин заводить».
2. **Свип цены ожидания (точка безубыточности).** Клиентский бенч по msleep-эндпоинту
   при задержке 0 / 1 / 5 / 20 / 100 мс: где async пересекает native. Сейчас тезис
   «граница применимости проходит по цене ожидания» доказывается косвенно — тут
   будет прямое измерение точки перелома. (Для БД то же самое даёт toxiproxy — см.
   lifecycle-план.)
3. **Свип размера payload.** selectMany/findOne с документами 1 КБ / 100 КБ / 1 МБ:
   оверхед msgpack+BSON растёт с размером данных, но в доке измерена только
   зависимость от числа вызовов.

## Честность

1. **Сравнение с нативной конкурентностью, а не только с последовательным кодом**
   (самое важное). В PHP она есть без SConcur: `curl_multi` для HTTP,
   `MYSQLI_ASYNC` + `mysqli_poll` для MySQL, `pg_send_query` для PostgreSQL. Сейчас
   async сравнивается с заведомо последовательным эталоном — скептик первым делом
   спросит про `curl_multi`. Даже проигрыш на голом HTTP в честной таблице сильнее,
   чем её отсутствие; козырь SConcur — единый API и смешанные фан-ауты (HTTP + SQL +
   Mongo в одном WaitGroup), чего у `curl_multi`/`mysqli_poll` нет.
2. **CPU-время (user+sys) на режим**, а не только wall-clock: веер прячет задержку,
   но платит процессором за конвертацию; сейчас эта цена видна только косвенно через
   `sync`.
3. **Холодный пул отдельной строкой.** В выводах доки упомянуто «3–15×», но самой
   цифры в таблицах нет: замер «первый веер после простоя vs прогретый» сделает
   утверждение проверяемым.
4. **Сброс mongo-коллекции между режимами.** У `updateMany`/`count` датасет растёт
   внутри прогона и native идёт первым по меньшей коллекции — сейчас это оговорка в
   доке; сброс между режимами уберёт её вовсе (частично закрывается
   lifecycle-планом).

## Трушность (близость к реальной жизни)

1. **Сравнение с экосистемными альтернативами**: amphp (revolt) и
   Swoole/OpenSwoole — реальные конкуренты за нишу «конкурентность в PHP»
   (RoadRunner — про другое, это process manager). Минимум: HTTP-клиент и один
   SQL-сценарий.
2. **SConcur внутри RoadRunner-воркера.** README уже говорит, что SConcur
   встраивается в любой долгоживущий процесс. Бенч «RR + SConcur в воркере» vs
   «RR + нативные драйверы» покажет историю «не заменить RR, а дополнить» — сильный
   сценарий для адопшена.
   **Сделано (2026-07-24):** ручка `/all-sconcur` в
   `tests/servers/roadrunner/rr-worker.php` (расширение в воркер — через
   `RR_WORKER_CMD`), замер `make bench-rr-sconcur-load-stats`; результат — +32% rps
   и p50 −25% против нативного RR (потолок задаёт модель «один запрос на воркер»),
   таблица и разбор в docs/benchmarks.ru.md, раздел «Сравнение с RoadRunner».
3. **Хвосты латентности под нагрузкой**: p99.9/max и гистограмма для `/all` — веер
   при 5 капнутых соединениях на процесс наверняка даёт очередь, и видно её именно
   в хвостах, а не в медиане.

## Приоритет
