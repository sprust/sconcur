# Неуловимый флаки-тест в полном сьюте

Статус: открыто, не воспроизводится по требованию. Наблюдение ведётся с
2026-07-04 (ветка `feature/optimization`).

## Симптом

`make test` (416 тестов) изредка падает ровно с одним фейлом:

- зелёный прогон: `Tests: 416, Assertions: 22511, Incomplete: 1`;
- флаки-прогон: `Tests: 416, Assertions: 22509, Failures: 1, Incomplete: 1` —
  на 2 assertion'а меньше, то есть тест падает до последних проверок.

Имя теста поймать не удалось: в testdox-хвосте вывода маркер не попался, а все
повторные прогоны (в том числе с `--stop-on-failure`) — зелёные.

## Хронология наблюдений (2026-07-04)

Всего ~10 полных прогонов за день, 2 фейла:

1. Первый фейл — первый `make test` сразу после `make ext-build` с буферизацией
   results-канала (коммит `78a9d06`) и заменой `uniqid` на счётчик. Немедленный
   повтор и ещё 2 прогона — зелёные.
2. Второй фейл — прогон вскоре после остановки/подъёма контейнера `sc-mysql`
   (проверка 500-логики `/all`, коммит `6f91811`) и убийства тестовых серверов
   в контейнере. Повтор — зелёный, затем 3 прогона подряд с
   `--stop-on-failure` — зелёные.

Общее у обоих случаев: фейл случался в первом прогоне после «встряски»
окружения (пересборка расширения / рестарт бэкенд-контейнера / фоновые
процессы в контейнере `php`). На «спокойной» машине не воспроизводится.

## Контекст ветки (что менялось незадолго до появления)

- `0ff717f` — suspend-first dispatch (cgo ушёл со стеков фиберов), дефолт
  `maxConcurrency = 0`. После этого коммита прогоны были зелёные (3 подряд).
- `78a9d06` — буферизация Go-канала результатов (1024) + дроп «протухших»
  результатов остановленных flow + чистка `pending` в `StopFlow`;
  `callbackKey` через счётчик. Первый фейл замечен в первом прогоне после
  этого коммита.

## Гипотезы (не проверены)

1. Тайминг-чувствительный серверный тест (shutdown/drain/timeout), которому
   стало «теснее» из-за фоновой нагрузки на хост — флаки был и мог быть до
   ветки, просто не попадался.
2. Следствие буферизации канала: изменившиеся тайминги доставки (результат
   лежит в буфере, а не в rendezvous) могли вскрыть гонку в каком-то тесте
   вокруг остановки flow. Если где-то «протухший» результат доезжает до PHP,
   это дало бы `FiberStateException: No coroutine for result` — в пойманных
   хвостах вывода такого не видно, но хвосты неполные.
3. Внешний фактор: бэкенд (mysql после рестарта, прогрев пула) отвечает
   ошибкой/медленно в одном тесте.

## Как ловить (план охоты)

1. Логировать имя упавшего теста персистентно: гонять сьют в цикле с
   `--log-junit /tmp/junit-$i.xml` (или `--log-events-text`), пока не поймается:

   ```bash
   i=0; while :; do i=$((i+1)); \
     docker compose exec -T php php -d extension=./ext/build/sconcur.so \
       vendor/bin/phpunit -d memory_limit=512M --colors=never \
       --log-junit /tmp/junit-$i.xml tests || break; done
   ```

2. Воспроизводить условия появления: прогон сразу после `make ext-build` /
   рестарта бэкенд-контейнера, при фоновой нагрузке на хост.
3. `--order-by=random` (с фиксацией seed при поимке) — расшатать
   меж-тестовые зависимости.
4. При поимке: если это `FiberStateException`/`UnexpectedTaskKeyException` —
   копать связку «буферизованный канал × StopFlow» (гипотеза 2); если таймаут
   серверного теста — гипотеза 1, поднять допуск в тесте.

## Критерий закрытия

Имя теста установлено, причина воспроизведена или опровергнута; либо 50+
полных прогонов подряд без фейла на спокойной машине (тогда закрыть как
внешний шум с пометкой здесь).
