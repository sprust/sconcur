# Батчинг переходов PHP ↔ Go и остаточные меры по границе

> **Статус: предложение (2026-07-22), не согласовано, работы не начаты.**
> Продолжение закрытой темы [`php-go-bridge-performance.md`](php-go-bridge-performance.md):
> тот документ — карта с профилем от 2026-07-02, закрывшим glue-идеи (msgpack-кодоген,
> слияние cgo-вызовов, pack/unpack в C — выигрыш < ~1–2% налога). Этот план берёт только
> то, что там осталось неопробованным и имеет измеренный или предполагаемый запас:
> батчинг результатов (идея F в узком виде), синхронный путь одним переходом (идея D +
> пункт роадмапа README) и точечные меры H/I. Всё остальное — см. «Не входит» ниже.

## Мотивация: где остался запас

По профилю 2026-07-02 (`php-go-bridge-performance.md`, разделы «Профиль» и
«PHP-профиль веера /all»):

- Потолок throughput `/all`: CPU/req ≈ 2029 мкс → ≈7900 rps на 16 воркеров, измерено
  ≈2580 (~⅓). Разница списана на «round-trip'ы `waitAny`/планировщика под конкуренцией» —
  единственная крупная статья, не разложенная до конца.
- Координация веера с приостановкой ≈ 515 мкс CPU на `/all` (~130 мкс на
  приостанавливающуюся корутину) — на порядок дороже чистой машинерии фиберов
  (~11 мкс/корутина).
- Синхронный путь: 3 cgo-перехода + одноразовый `Flow` + горутина + канал на каждый
  вызов; потолок экономии по Go-бенчу < ~8 мкс/вызов (идея D). Пункт роадмапа README
  («Optimize synchronous-mode execution») до сих пор не реализован.

**Важная оговорка о свежести цифр.** После того профиля общий канал результатов стал
буферизованным (`resultsBufferSize = 1024`, `ext/internal/handler/handler.go:23`) —
именно чтобы убрать rendezvous «две futex-побудки на результат», который тогда
доминировал в цене координации. Цифры 515/130 мкс и разрыв ⅓ потолка могли измениться.
Поэтому план начинается с обязательной повторной атрибуции (фаза 0), и каждая
последующая фаза гейтится её результатом — как раньше замеры закрыли идеи B/C.

## Фаза 0 — повторная атрибуция (гейт для всего остального)

Разложить сегодняшнюю цену одного результата, прошедшего через `waitAny`, на статьи:

1. Go-сторона: `Handler.WaitAny` при заполненном буфере (быстрый неблокирующий путь)
   vs пустом (парковка + побудка) — Go-бенч без cgo (`CGO_ENABLED=0`), по образцу
   бенчей из прошлого профиля.
2. Кадр + cgo: `buildResultFrame` + `C.CBytes` (`ext/main.go:48`, `ext/main.go:82`) +
   копия `RETVAL_STRINGL` (`ext/sconcur.c:192`) — микробенч цикла `waitAny` из PHP на
   заранее готовых N результатах (веер sleeper'ов, дождаться завершения всех задач,
   затем мерить чистый цикл выборки).
3. PHP-сторона на результат: `parseWaitResponse` + `TaskResultDto`
   (`src/Connection/Extension.php:201`) отдельно от `resumeByResult` (фибер + `State`,
   `src/Scheduler/Scheduler.php:460`).
4. Под конкуренцией: профиль (`perf` для PHP, pprof для Go) на `make bench-*` для
   `socket-throughput` (~150k rt/s) и на `/all` (`bench-http-load-stats`) — куда
   уходит разрыв до расчётного потолка.

Критерий: фазы 1–2 делаются, только если их адресуемая статья (пп. 1–2 + вызов
userland-функции на результат) составляет заметную долю (ориентир — ≥ ~10% цены
round-trip'а под конкуренцией). Иначе — закрыть их так же, как были закрыты B/C, и
дополнить карту в `php-go-bridge-performance.md`.

## Фаза 1 — `waitAnyBatch`: дренаж готовых результатов одним переходом

Сейчас каждый готовый результат — отдельный переход: `Scheduler::tick()` →
`Extension::waitAny()` → один кадр (`src/Scheduler/Scheduler.php:452`,
`ext/main.go:164`). При широком веере и на серверном цикле в буфере канала обычно
лежит несколько готовых результатов — их можно забирать пачкой.

### Go

- `Handler.WaitAnyBatch(max int) ([]*dto.Result, error)`: сначала `popAnyPending`,
  затем блокирующий приём первого результата (как сейчас в `WaitAny`,
  `ext/internal/handler/handler.go:62`), затем неблокирующий `select`-дренаж канала до
  `max` штук; каждый результат проходит `deliver()` (учёт + фильтр результатов
  остановленных flow, `ext/internal/handler/handler.go:152`).
- `Handler.WaitAnyTimeoutBatch(ms, max int)` — то же с дедлайном для серверного цикла;
  сентинел `timeout` сохраняется.
- `ext/main.go`: экспорты `waitAnyBatch(max)` и `waitAnyTimeoutBatch(ms, max)`.
  Формат мультикадра: `[count uint16][len uint32][кадр]...` — внутри каждый кадр
  в текущем формате `buildResultFrame`, без изменений.

### C-глю и PHP

- `ext/sconcur.c`: две новые функции (arginfo + регистрация), возврат
  `RETVAL_STRINGL` как у `waitAny`.
- `Extension::waitAnyBatch()` / `waitAnyTimeoutBatch()`: цикл по мультикадру, каждый
  кадр — через существующий `parseWaitResponse`; возврат `list<TaskResultDto>`.
- `Scheduler`: локальная очередь готовых результатов. `tick()` берёт из очереди, при
  пустой — пополняет батч-вызовом; `serve()` (`src/Scheduler/Scheduler.php:344`)
  так же, сохраняя текущую логику по каждому событию (re-arm `next()` до `spawn`,
  ветка draining).
- Старые `waitAny`/`waitAnyTimeout` остаются в расширении (их использует
  `Extension::waitAny` для совместимости тестов), но планировщик переходит на батч.

### Риски и тонкости

- **Устаревшие результаты в локальной очереди PHP** — главный риск корректности.
  Обработка результата i из пачки может остановить flow результата j из той же пачки
  (`WaitGroup::stop`, shutdown, серверный drain). Go-сторона такие результаты фильтрует
  на выдаче (`deliver`), но пачка выдаётся до остановки. Сейчас `resumeByResult`
  бросает `FiberStateException`, если фибер не найден (`src/Scheduler/Scheduler.php:467`)
  — с очередью «ничей» результат становится легитимным. Решение: при выборке из
  очереди результат, чей flow уже удалён из `State`, молча дропать (понадобится
  проверка вида `State::hasFlow()`); строгий инвариант сохранить для результатов,
  полученных напрямую.
- `max` пачки — небольшая константа (например, 64): ограничивает пиковый размер
  мультикадра и время между проверками `shouldStop()` в серверном цикле.
- Протокол меняется → **бамп минорной версии** (0.7.0 → 0.8.0) во всех трёх
  источниках (`ext/main.go` `version()`, `Extension::REQUIRED_EXTENSION_VERSION`,
  `composer.json`), один раз на ветку; `VersionConsistencyTest` следит.

### Ожидаемый эффект и мерило

Экономится на каждый результат после первого: cgo-переход, сборка+две копии кадра,
вызов userland-функции. Это единицы мкс на результат — эффект ожидаем на
высокочастотных сценариях: `socket-throughput` (~150k rt/s), `ws-throughput`
(~113k rt/s), пустая ручка HTTP (~67.6k rps), `/all` rps. Мерить именно их до/после
(медиана из 3, методология `docs/benchmarks.md`); DB-бенчи `bench-all` — контроль
отсутствия регресса.

## Фаза 2 — синхронный путь одним переходом (`pushWait`, идея D + роадмап README)

Сейчас sync-вызов вне фибера: `push` + `wait` + `stopFlow` — 3 перехода, одноразовый
`Flow` + канал + горутина на каждый вызов (`src/Features/FeatureExecutor.php:41`,
`src/Features/FeatureExecutor.php:112`).

- Новый экспорт `pushWait(fk, mt, tk, payload)`: исполняет обработчик фичи **инлайн на
  cgo-нити** — без `Flow`, канала и горутины — и сразу возвращает кадр результата.
  Для PHP-стороны семантика не меняется: sync-вызов и так блокирует поток на `wait`.
- Стриминговые результаты (`hasNext`): обработчик регистрирует flow только когда
  результат стримовый — первый батч возвращается из `pushWait`, продолжение идёт по
  текущему пути `next`/`wait`. Нестримовый вызов не создаёт flow вовсе.
- PHP: sync-ветки `FeatureExecutor::exec`/`next` переходят на `pushWait`; `handleSync`
  упрощается (нет `stopFlow` для нестримового случая). Async-путь не трогается.
- Потолок по старому Go-профилю: < ~8 мкс/вызов (fresh−reused flow ≈ 8 аллокаций /
  ~800 Б) плюс два сэкономленных перехода (~0.2 мкс) плюс PHP-обвязка. Скромно, но
  это заявленный пункт роадмапа README, и правка локальна для sync-пути.
- Требования фич сохраняются: контекст-отмена и дедлайн работают как раньше —
  обработчик получает тот же `context` задачи; отличие только в том, где он исполняется.
- Тот же бамп версии (в рамках той же ветки — без повторного бампа).

Гейт: делать после фазы 0, если sync-путь вообще в приоритете (его налог виден только
на операциях с околонулевой задержкой — см. вывод закрытого плана).

## Фаза 3 — точечные меры по результатам профиля (опционально)

- **H — ленивый/типизированный декод результата**: если профиль фазы 0 подтвердит, что
  на дешёвых операциях доминирует сборка zval (жадный `msgpack_unpack` больших
  вложенных массивов), — декодировать поля по обращению; для выборок — стриминг с
  ленивым декодом (частично уже есть через `next()`).
- **I — считать на стороне Go**: обобщить паттерн download (`HttpClient::download` —
  тело не заходит в PHP) на случаи, где PHP нужен только итог (количество, статус,
  агрегат): считать в Go, возвращать маленький результат.

Обе меры — отдельные точечные PR по конкретному профилю, не «переделка транспорта».

## Не входит (закрыто замерами 2026-07-02 — не переоткрывать без новых данных)

| Идея | Почему закрыта |
| --- | --- |
| Кодоген/ручной msgpack в Go (`tinylib/msgp`) | unmarshal ≈ 0.19 мкс/вызов — не окупает кодоген |
| Слияние cgo-вызовов per-op (`pushWait` ради числа переходов как такового) | переход ≈ 0.082 мкс; в фазе 2 сливание — побочный бонус, выигрыш там от отказа от Flow/горутины |
| pack/unpack в C через C-API ext-msgpack | userland pack+unpack ≈ 0.28 мкс (~1% налога) + хрупкая зависимость |
| Замена MessagePack (FlatBuffers/Cap'n Proto/Protobuf) | PHP-реализации pure-PHP медленнее C-шного ext-msgpack; кодоген + переписывание payload'ов не оправданы |
| FFI вместо C-расширения | FFI-вызов дороже zend-функции |
| Shared memory / ring buffer (E) | cgo не доминирует; риск/сложность несопоставимы с выигрышем |
| zval-мост / zero-copy payload | разобрано и отклонено в закрытом плане (жизненный цикл zval, анти-паттерн cgo) |

## Порядок работ, гейты, процесс

1. **Фаза 0** — атрибуция; по её итогам зафиксировать в этом файле решение по фазам 1–2
   (по образцу таблицы статусов закрытого плана).
2. **Фаза 1** — `waitAnyBatch` (если прошла гейт): отдельный измеримый PR с бампом
   0.8.0.
3. **Фаза 2** — `pushWait` (если прошла гейт): отдельный PR, та же версия в рамках
   одной ветки.
4. **Фаза 3** — по профилю, точечно.

Каждый шаг: согласование с мейнтейнером до реализации (правки ядра и протокола),
`make check` (cs-fixer, phpstan, tests, ext-test), бенчмарки до/после
(`make bench-all`, `bench-socket-throughput`, `bench-ws-throughput`,
`bench-http-load-stats`), медиана из 3 прогонов, контроль отсутствия регресса
async-пути — главной ценности библиотеки.
